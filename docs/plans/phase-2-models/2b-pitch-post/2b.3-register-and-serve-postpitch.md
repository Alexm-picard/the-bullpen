# 2b.3 — Register + serve post-pitch model

> Owning phase: `phase-2-models/2b-pitch-post` · Estimated effort: `4–6 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:

- ONNX export of `pitch_outcome_post/v1`
- Spring serving via the same `POST /v1/predict/pitch` endpoint, dispatched by a `head` query parameter (`?head=pre|post`)
- `PitchInferenceService` extended to load both models; route based on `head`
- Java parity tests for 10 post-head fixtures
- Hard rule (decision `[35]`): two separate models in the registry, NOT one model with feature masking

**OUT of scope**:

- Soft-cut decision: if Phase 2 schedule is at risk by end of Wk 12, this whole sub-tree (2b) is the first soft-cut candidate. If cut, `?head=pre` becomes the only option.

---

## Objectives

1. `POST /v1/predict/pitch?head=post` returns 5-class probabilities using post-pitch features in <100 ms p95.
2. Java parity tests pass for 10 post-head fixtures within `1e-6`.
3. `head=pre` and `head=post` are dispatched to different ONNX sessions, no cross-contamination.

---

## Dependencies

**Upstream**: `2b.2`, `2a.8` (serving plumbing).

## Required files / modules

**New**:

- `ml/thebullpen/pitch/export_post_onnx.py`
- `ml/tests/fixtures/parity_pitch_post_001.json` + expected (×10)
- `backend/src/main/java/net/thebullpen/baseball/inference/FeaturePipelinePitchPost.java`
- `backend/src/test/java/net/thebullpen/baseball/inference/PitchPostParityTest.java`

**Modified**:

- `PredictPitchController` — adds `@RequestParam(defaultValue="pre") String head` and dispatches.
- `PitchRequest` — extended with optional Tier 4 fields (validated when `head=post`).
- `PitchInferenceService` — holds two `OnnxModel` instances and two `IsotonicCalibratorJava` instances.

---

## Step-by-step implementation tasks

1. Author `export_post_onnx.py` mirroring 2a.8.
2. Generate 10 post-head parity fixtures.
3. Author `FeaturePipelinePitchPost`: same structure as `FeaturePipelinePitchPre` but with Tier 4 columns appended.
4. Modify `PitchRequest`:
   ```java
   public record PitchRequest(
     // Tier 1+2+3 fields
     ...,
     // Tier 4 (only required when head=post)
     @Nullable String pitchType,
     @Nullable Float releaseSpeedMph, ...
   ) {}
   ```
   Add a Spring validator: when `head=post`, all Tier 4 fields must be non-null.
5. Modify `PitchInferenceService`:
   ```java
   public PitchPredictionResponse predict(PitchRequest req, Head head) {
     OnnxModel model = head == Head.PRE ? preModel : postModel;
     IsotonicCalibratorJava cal = head == Head.PRE ? preCal : postCal;
     FeaturePipelinePitch pipeline = head == Head.PRE ? prePipeline : postPipeline;
     // ... features → ONNX → calibrator → response
   }
   ```
6. Modify `PredictPitchController` to dispatch.
7. Author `PitchPostParityTest`.
8. Update OpenAPI; regen frontend types. Frontend hook `usePredictPitch` accepts `{ head: 'pre' | 'post' }`.

## Acceptance criteria

- [ ] `POST /v1/predict/pitch?head=post` returns 200 in <100 ms p95.
- [ ] `head=post` without Tier 4 fields → 400.
- [ ] 10 parity fixtures pass.
- [ ] `head=pre` continues to work unchanged.
- [ ] Both ONNX sessions warm up at startup; readiness gate accounts for both.
- [ ] OpenAPI spec includes `head` enum.

---

## Known edge cases

- **`head=pre` is the default**: keeps clients that don't know about post head working.
- **Memory**: two ONNX sessions ≈ 200-400 MB extra. API memory cap (4G in 0.3) is fine.
- **Soft-cut path**: if 2b is cut, leave `head` query param accepting `pre` only and document the cut in `00-MASTER.md` Status Log.
- **Mixed-head A/B**: not allowed in v1. Each head A/Bs against itself only (Phase 3b will enforce via routing config keyed by `model_name`).
