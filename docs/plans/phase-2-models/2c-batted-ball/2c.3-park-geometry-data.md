# 2c.3 — Park geometry data

> Owning phase: `phase-2-models/2c-batted-ball` · Estimated effort: `8–12 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Per-park geometry data for all 30 MLB parks:
  - Outfield wall polyline (sequence of `(distance_ft, height_ft)` points along the fence from LF foul pole to RF foul pole)
  - Foul territory area + dimensions
  - Stand height behind the fence (for HR vs out determination on fly balls)
  - Park altitude, default atmosphere (default temperature, pressure, humidity used when no weather data)
  - Home plate orientation (the angle of the field's centerline from true north — affects wind direction effect)
- Stored as `infra/park_geometry/<park_id>.json`
- A small Python loader returning a `ParkGeometry` dataclass

**OUT of scope**:
- Roof state for retractable-roof parks (TOR, MIA, AZ, etc.) — v1 treats them as always open with clear-sky atmosphere; v1.5 may improve
- Stadium-relocation history (Athletics, Rays) — single park per `park_id` for v1

## Objectives

1. 30 park JSON files exist, each schema-validated.

3. Sanity tests: a flyball that lands 320 ft at 12 ft height in left-center is correctly classified as HR at Yankee Stadium and out at Comerica Park (manual sanity).

## Step-by-step implementation tasks

1. Author the schema:
   ```json
   {
     "park_id": "string (3-char team code)",
     "name": "string",
     "altitude_ft": "number",
     "default_atmosphere": { "temp_C": "number", "pressure_hPa": "number", "humidity_pct": "number" },
     "centerline_bearing_deg": "number (0=N, 90=E)",
     "fence_polyline": [
       { "angle_from_centerline_deg": -45, "distance_ft": 318, "height_ft": 8 },
       ...
     ],
     "behind_fence_stand_height_ft": "number (default 25)",
     "foul_territory_sqft": "number"
   }
   ```
2. Source the data from MLB published park dimensions + Andrew Clem's stadium diagrams + Baseball Reference. Manually curate. ~20 min/park × 30 ≈ 10 hours.
3. Author `loader.py`: validates each JSON against schema, returns `ParkGeometry` dataclass with interpolation helper for fence height at arbitrary spray angles.
4. Author `_classify.py`:
   ```python
   def classify_outcome(traj: Trajectory, park: ParkGeometry, batter_runs_speed_fps: float) -> Outcome:
       # 1. Find where traj crosses ground (z=0) — if NaN, ball is still in air.
       # 2. If landing point is outside fence polyline at corresponding angle, AND ball was above fence height when crossing the fence: HR.
       # 3. Otherwise: ball is in field of play; outcome = single/double/triple/out depends on landing distance, hang time, batter speed (heuristic).
       ...
   ```
   The 5-class outcome (out/single/double/triple/HR) is what 2c.4 will retrodict.
5. Author tests: `test_loader.py` validates all 30 files load; `test_classify.py` covers known examples (Aaron Judge HR @ NYA = HR; same trajectory @ COL = HR with bigger margin; line drive in foul territory = foul).
6. Sanity check: a 380-ft fly ball at 25 ft of height to right-center is HR at PIT (where RF is short) but out at MIN's old Met (deep RC).

## Known edge cases

- **The Cathedral / Fenway / Coors "weird" walls**: high parts of the fence (Green Monster, Coliseum tarps) require dense polyline points. Use ~12 polyline points for those parks.
- **Foul poles**: balls hitting just inside the foul pole are HR. Approximated via polyline; small angular tolerance (±0.5°) treats as HR.
- **Roof state**: covered roof = clear sky atmosphere, no wind. v1 simplification.
- **Multi-tier fences (BAL warehouse, BOS Monster Seats)**: just use the lowest height that still results in HR.
- **`batter_runs_speed_fps`**: comes from Statcast `sprint_speed`. For 2c.4 retrodiction, use league-mean (~27 fps) since per-batter speed isn't a 2c feature.
