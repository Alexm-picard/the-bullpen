# 2a.9 — Forward simulator (analytical + Monte Carlo)

> Owning phase: `phase-2-models/2a-pitch-pre` · Estimated effort: `6–8 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Java forward simulator deriving plate-appearance statistics from per-pitch model outputs (decision `[53]`)
- Two implementations (decision `[54]`):
  - Analytical: 15×15 transition matrix Q + absorbing matrix R; expected steps via fundamental matrix `(I − Q)⁻¹`
  - Monte Carlo: N=10,000 simulated PAs from a starting count; full distribution of PA-length, P(BB), P(K), P(BIP)
- `POST /v1/simulate/plate-appearance` Spring endpoint (analytical, production)
- `POST /v1/simulate/plate-appearance/monte-carlo` (diagnostic)
- Convergence test: MC mean → analytical within tolerance

**OUT of scope**:
- Half-inning extension → v1.5 (decision `[55]`)
- Frontend visualizations using this → Phase 4

---

## Objectives

1. Analytical endpoint returns E[pitches per PA], P(BB), P(K), P(BIP) in <50 ms.
2. Monte Carlo (N=10,000) endpoint matches analytical mean within ±0.5% for 100 random starting counts.
3. Both implementations are deterministic given a seed (MC uses it).
4. Markov-chain code is unit-tested against hand-computed small examples.

## Step-by-step implementation tasks

1. Author `PlateAppearanceMarkov`:
   - 12 transient states: `(balls, strikes)` for `0..3 × 0..2` minus terminal counts.
   - 3 absorbing states: `BB`, `K`, `BIP`.
   - Transition function: given pitcher's per-pitch probabilities `{ball, called_strike, swinging_strike, foul, in_play}`:
     - `ball` → state with `balls+1` (or BB if `balls=3`)
     - `called_strike` or `swinging_strike` → `strikes+1` (or K if `strikes=2`)
     - `foul` → `strikes+1` if `strikes < 2`, else stays at `strikes=2`
     - `in_play` → BIP
   - Build a 15×15 transition matrix.
2. Author `AnalyticalSolver`:
   - Decompose into Q (12×12, transient→transient) and R (12×3, transient→absorbing).
   - Fundamental matrix `N = (I − Q)⁻¹`.
   - Expected number of steps from state `s` = sum of row `s` in N.
   - Absorption probability matrix `B = N · R`.
3. Author `MonteCarloSimulator`:
   - Given start state and per-pitch probability vector function (which can vary per state), roll N=10,000 PAs.
   - Each roll: sample next event from the per-pitch distribution, transition state, count pitches, terminate at absorbing state.
   - Returns histogram + summary stats.
4. Author `SimulateController` exposing both endpoints. Same per-pitch probability function (calls `PitchInferenceService`).
5. Author `MarkovTest`:
   - Hand-construct a 2-state matrix; assert N = (I − Q)⁻¹ matches.
   - Construct a deterministic chain (always called_strike); assert E[steps] = 3.
6. Author `AnalyticalVsMonteCarloTest`:
   - 100 random pitcher probability vectors.
   - For each: compute analytical and MC; assert MC mean within ±0.5% of analytical.

## Known edge cases

- **Foul-with-2-strikes loop**: the chain has a self-loop; expected steps converge mathematically because the foul probability < 1 (rarely 1 in real data).
- **Numerical singularity**: `(I − Q)⁻¹` exists iff Q is sub-stochastic with at least one absorbing path reachable. League-average probabilities always satisfy this. Catch `RuntimeException` from inversion and surface as 500 with `error.code=singular_matrix` (theoretically unreachable).
- **Per-pitch probabilities depend on count**: in real life, pitchers throw differently in 0-0 vs 3-2. The simulator must call `PitchInferenceService` PER STATE, not once. Document.
- **Determinism**: MC uses `SplittableRandom(seed)`; same seed → same trajectory.
- **`PitchRequest` constructor**: building 12 different requests for 12 states is redundant. Cache the per-state probabilities once per simulation.
