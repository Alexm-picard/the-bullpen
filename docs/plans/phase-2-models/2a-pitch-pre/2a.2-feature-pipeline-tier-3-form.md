## Scope boundaries

**IN scope**:

- Tier 3 features computed via streaming temporal cutoff:
  - `pitcher_pitches_last_28d`, `pitcher_pitches_in_game`, `days_since_last_appearance`
  - `pitcher_strike_rate_28d`, `pitcher_swstrike_rate_28d`, `pitcher_inplay_rate_28d`
  - `batter_inplay_rate_28d`, `batter_strikeout_rate_28d`, `batter_walk_rate_28d`
  - `season_to_date` variants of the above (cutoff at `game_date - 1`)
- Stream computation: rows ordered by `(entity_id, game_date, at_bat_index, pitch_number)` with running counters reset on new entity
- New columns added to `features` via additive migration
- Tests using a small synthetic stream that reproduces hand-computed counters

**OUT of scope**:

- Catcher/umpire/sequence/weather/slumps — `[41]` defers all to v1.5
- Tier 4 post-pitch features → Phase 2b
- Java reproduction of Tier 3 — 2a.8 only needs _current-state_ values, not the streaming logic; document the simplification

2. By construction: every Tier 3 value uses only data with `as_of_date < pitch_date`.
3. Synthetic-stream tests demonstrate counter resets per pitcher and per game.
4. End-to-end run on full corpus ≤ 4 hours wall time.

## Step-by-step implementation tasks

1. `V006__features_tier3.sql` — `ALTER TABLE features ADD COLUMN ...` for each Tier 3 field, all `Nullable(Float32)` or `Nullable(UInt32)`.
2. Author `tier_3_form.py`:
   - For each pitch row, compute counters from `pitches` rows WHERE `(pitcher_id=p AND game_date BETWEEN pitch_date - 28 AND pitch_date - 1)` etc.
   - Implement as a single SQL aggregation per fold (cleaner) OR Python streaming (slower but easier to test). Choose SQL.
   - Sample SQL pattern for `pitcher_strike_rate_28d`:
     ```sql
     SELECT p.game_id, p.at_bat_index, p.pitch_number,
       (SELECT countIf(description IN ('called_strike','swinging_strike','foul'))
                / nullIf(count(), 0)
        FROM pitches WHERE pitcher_id = p.pitcher_id
          AND game_date >= p.game_date - 28 AND game_date < p.game_date) AS strike_rate
     FROM pitches p WHERE ...
     ```
   - For "in-game" counters, use `runningSum` over an `ORDER BY` window.
3. Persist results back via `INSERT INTO features (... tier3 cols ...) SELECT ...` using `JOIN` on the PK.
4. Author `test_tier_3_form.py`: synthetic 30-row stream with known counter values; assert exact match per row.
5. Run per fold; verify nulls drop to 0 after warm-up window (~28 days into 2015 data).
6. Reconcile with leakage tests in 2a.3 — Tier 3 must pass `test_calendar_date_trace.py` for 10 random pitches.

## Known edge cases

- **Mid-season pitcher debut**: `days_since_last_appearance` is undefined for first appearance — use `NULL`, not 0.
- **Doubleheaders**: same `pitcher_id` + same `game_date` in two `game_id`s. Counters must recognize game boundaries via `(pitcher_id, game_id)` not just `(pitcher_id, game_date)`.
- **In-game counter at pitch 1**: should be 0, not NULL.
- **Cross-season boundary**: `season_to_date` resets at the season boundary; pitchers traded mid-season keep one rolling history (not split per team in v1).
- **`description` filter**: only count pitches with `description != 'unknown'` for rate calculations.
