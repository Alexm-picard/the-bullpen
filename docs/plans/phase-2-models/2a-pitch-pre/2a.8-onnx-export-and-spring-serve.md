# 2a.8 — ONNX export + Spring serve `/v1/predict/pitch`

> Owning phase: `phase-2-models/2a-pitch-pre` · Estimated effort: `8–10 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:

- LightGBM (`pitch_outcome_pre/v1`) → ONNX export with the same numeric-preprocessing-in-graph + categorical-lookups-in-Java boundary established in 1.4
- `feature_pipeline.json` for the production pitch model — extends 1.4's toy version with Tier 1+2+3 columns
- `park_hr_rate.json`-style lookups: TE lookups for pitcher_id, batter_id (per fold; v1 uses fold-4 for production); category mappings
- Spring `PredictPitchController` exposing `POST /v1/predict/pitch` with full request DTO (count, base_state, pitcher_id, batter_id, etc.)
- `PitchInferenceService` loading model + calibrator + lookups from disk
- Feature-pipeline parity test extended for the production pitch model
- Live ECE check: post-deploy smoke fires N predictions on a 2025 holdout fixture; calibration drift > 1.5× training fails the deploy

**OUT of scope**:

- Registry registration of this model — Phase 3a (this leaf hardcodes paths)
- Shadow-mode routing — Phase 3b
- Post-pitch head — Phase 2b

## Step-by-step implementation tasks

1. Author `export_pre_onnx.py` (Python): `convert_lightgbm` with `FloatTensorType([None, len(FEATURES)])`. Save `model.onnx`.
2. Generate 10 parity fixtures: random pitches from 2025 → preprocessed feature vector + expected probs → save under `ml/tests/fixtures/`.
3. Author `IsotonicCalibratorJava`: parses `calibrator.json` per class; applies via piecewise-linear interpolation; renormalizes.
4. Author `FeaturePipelinePitchPre`:
   - Reads `feature_pipeline.json` for column order and TE lookup file paths.
   - Loads `pitcher_te_*.json`, `batter_te_*.json` into hash maps at startup.
   - `transform(PitchRequest, currentTier3Snapshot) -> float[N]`:
     - Tier 1 from request
     - Tier 2 from request lookups
     - Tier 3 from `features`-table snapshot (fetched per request OR pre-computed nightly per pitcher_id)
5. **Tier 3 strategy decision** (document in module): for live serving, Tier 3 values are looked up from a denormalized table `pitcher_form_current` populated nightly by the worker. Avoids streaming computation in the request path. Document trade-off: form lag = 1 day.
6. Add migration `V007__pitcher_form_current.sql`:
   ```sql
   CREATE TABLE pitcher_form_current (
     pitcher_id UInt32,
     as_of_date Date,
     pitches_in_game UInt32,            -- not available pre-game; default 0
     pitches_last_28d UInt32,
     strike_rate_28d Float32,
     swstrike_rate_28d Float32,
     inplay_rate_28d Float32,
     days_since_last_appearance Nullable(UInt16)
   ) ENGINE = ReplacingMergeTree(as_of_date)
   PRIMARY KEY (pitcher_id);
   ```
   Worker job populates this nightly (Phase 3 job; this leaf leaves a TODO if the job isn't ready yet — fall back to computing inline at request time).
7. Author `PitchInferenceService`:
   - On startup: load ONNX model, calibrator, feature pipeline, lookups. Warm up with 3 dummy predictions.
   - `predict(PitchRequest)`: features → ONNX → calibrator → return ranked class probabilities + winner class.
8. Author `PredictPitchController` `@PostMapping("/predict/pitch")`. Bean Validation. Returns `PitchPredictionResponse { probs: { ball, called_strike, ... }, modelVersion, latencyMicros, traceId }`.
9. Author `PredictPitchControllerTest` (MockMvc): 200 happy path + 400 validation paths.
10. Author `PitchPreParityTest`: load 10 parity fixtures; run through Java pipeline; compare against `..._expected.json`. All within `1e-6`.
11. Wire into the async logger from 1.7 — log every prediction.
12. Update OpenAPI; regen frontend types.

## Known edge cases

- **Tier 3 lag**: 1-day stale during in-game play. Acceptable for v1; document. Post-pitch head (2b) has same constraint.
- **`pitches_in_game`**: served as 0 for live games until Phase 4d's live polling populates it. Pre-game prediction is the use case for v1.
- **Cold pitchers / batters**: TE lookup falls back to global prior. Java pipeline must implement the same fallback as Python.
- **Calibrator monotonicity**: as in 2a.5; renormalization can break per-class monotonicity. Java version must match Python's exact arithmetic.
- **Feature schema hash mismatch**: this leaf precedes 3a.3 enforcement. Once 3a.3 lands, the registration step will validate this hash matches the production pipeline.
