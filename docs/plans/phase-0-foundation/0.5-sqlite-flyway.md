# 0.5 — SQLite + Flyway

> Owning phase: `phase-0-foundation` · Estimated effort: `3–4 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- SQLite file at `/var/lib/thebullpen/registry.sqlite` (created by Flyway baseline)
- `xerial sqlite-jdbc` dependency in `backend/pom.xml`
- Flyway runs SQLite migrations on Spring startup from `backend/src/main/resources/db/migration/sqlite/`
- `V001__baseline.sql` creates a marker table only (real registry schema lands in 3a.1)
- A trivial repository (`SqliteHealthRepository`) that performs `SELECT 1` confirming the connection
- Integration test: embedded SQLite (in-memory) loads the V001 migration cleanly
- File permissions: `0640 thebullpen:thebullpen`

**OUT of scope**:
- `model_versions`, `model_routing`, `experiment_results`, `retraining_queue` — those land in Phase 3
- WAL tuning beyond the default — revisit if registry hot-paths emerge

---

## Objectives

1. SQLite file exists, Flyway records baseline migration.
2. Spring connects on startup; failure is loud (Spring fails to start, not silently degraded).
3. Flyway migration directory and version table are in place for future leaves.
4. Tests use embedded SQLite and never touch the production file.

## Step-by-step implementation tasks

1. Add to `backend/pom.xml`:
   ```xml
   <dependency>
     <groupId>org.xerial</groupId>
     <artifactId>sqlite-jdbc</artifactId>
     <version>3.46.x</version>
   </dependency>
   <dependency>
     <groupId>org.flywaydb</groupId>
     <artifactId>flyway-core</artifactId>
   </dependency>
   ```
2. Add to `application.yml`:
   ```yaml
   bullpen:
     sqlite:
       url: jdbc:sqlite:/var/lib/thebullpen/registry.sqlite
   spring:
     flyway:
       url: ${bullpen.sqlite.url}
       locations: classpath:db/migration/sqlite
       baseline-on-migrate: true
       table: flyway_schema_history
       schemas: main
   ```
3. Author `SqliteDataSourceConfig` returning a HikariDataSource pointed at the file. Set `journal_mode=WAL`, `synchronous=NORMAL`, `foreign_keys=ON` via init SQL.
4. Author `V001__baseline.sql`:
   ```sql
   CREATE TABLE IF NOT EXISTS bullpen_meta (
       key TEXT PRIMARY KEY,
       value TEXT NOT NULL,
       updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
   );
   INSERT INTO bullpen_meta (key, value) VALUES ('schema_baseline', '1');
   ```
5. Author `SqliteHealthRepository.ping()` doing `SELECT 1` against the registry datasource.
6. Add a one-liner `@PostConstruct` in `SqliteDataSourceConfig` that runs `ping()` and logs the result. Failure must abort startup.
7. Author `SqliteFlywayIT` using `:memory:` JDBC URL: assert V001 applied, `bullpen_meta` row exists.
8. Run `./mvnw verify`. Migration runs against in-memory.
9. Deploy: rebuild JAR, `systemctl restart thebullpen-api`. Verify file exists:
   ```bash
   ls -la /var/lib/thebullpen/registry.sqlite
   sqlite3 /var/lib/thebullpen/registry.sqlite "SELECT * FROM bullpen_meta;"
   ```
10. Permissions: `sudo chown thebullpen:thebullpen /var/lib/thebullpen/registry.sqlite && sudo chmod 0640 /var/lib/thebullpen/registry.sqlite`.

## Acceptance criteria (Definition of Done)

- [ ] `/var/lib/thebullpen/registry.sqlite` exists, mode `0640 thebullpen:thebullpen`.
- [ ] `sqlite3 /var/lib/thebullpen/registry.sqlite "SELECT * FROM flyway_schema_history;"` shows V001 applied.
- [ ] `SELECT * FROM bullpen_meta WHERE key='schema_baseline';` returns row with value `1`.
- [ ] Spring API + Worker both start cleanly; `journalctl` shows `Flyway: Successfully applied 1 migration`.
- [ ] Re-starting the JAR does NOT re-apply V001 (Flyway is idempotent).
- [ ] All declared tests pass locally and in CI.
- [ ] Restart-after-`wsl --shutdown` preserves the file and migration state.

## Known edge cases

- **WAL files**: `registry.sqlite-wal` and `-shm` may appear. Don't back them up separately — checkpoint via `PRAGMA wal_checkpoint(TRUNCATE)` in the backup script (Phase 0.10).
- **`baseline-on-migrate`**: needed because the file may exist from a manual debug session. Set true so Flyway doesn't refuse.
- **Concurrent writers**: Spring API + Worker both connect. SQLite WAL handles this fine for our load — but if both run a write `UPDATE` simultaneously you can hit `SQLITE_BUSY`. Set `busy_timeout=5000` in init SQL.
