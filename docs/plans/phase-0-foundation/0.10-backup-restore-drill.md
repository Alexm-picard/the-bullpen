# 0.10 — Backup + restore drill ★ Phase 0 exit gate

> Owning phase: `phase-0-foundation` · Estimated effort: `5–7 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- `clickhouse-backup` installed and configured for nightly local snapshots
- `rclone` configured with Backblaze B2 credentials, copying snapshots off-machine
- Retention 7-4-12 (daily / weekly / monthly) implemented via `clickhouse-backup` lifecycle + a small bash/python pruner
- `backup.sh` orchestrator + systemd timer at 5 AM ET
- SQLite registry file backed up alongside (with `PRAGMA wal_checkpoint(TRUNCATE)` first)
- Model artifacts directory `/var/lib/thebullpen/models/` mirrored weekly to B2
- **Restore drill executed once**: pull yesterday's backup, restore to a fresh ClickHouse, verify counts match
- Runbook `ops/runbooks/restore-drill.md` written and demonstrably correct

**OUT of scope**:
- Backups of Prometheus TSDB — not critical; can rebuild from scratch
- Cross-region replication — single B2 bucket is enough
- Encryption beyond B2's at-rest — fine for this data

---

## Objectives

1. Daily backup at 5 AM ET runs unattended.
2. Off-machine copy in B2 within 30 min of local snapshot.
3. Restore drill executed end-to-end: a backup taken today is restored to a fresh ClickHouse instance and `count(*)` from each major table matches expectations.
4. Retention enforced: counting files in B2 shows exactly 7 daily, 4 weekly, 12 monthly after 12 months of rolling.
5. Discipline rule 7 (CLAUDE.md) satisfied: untested backups don't count → drill done.

## Step-by-step implementation tasks

1. Create B2 account + bucket (`thebullpen-backups`). Generate an application key scoped to the bucket. Copy ID + key.
2. Append to `/etc/thebullpen/secrets.env`:
   ```
   B2_ACCOUNT_ID=...
   B2_APPLICATION_KEY=...
   B2_BUCKET_NAME=thebullpen-backups
   ```
3. Install dependencies inside WSL:
   ```bash
   sudo apt-get install -y rclone
   curl -fsSL https://github.com/Altinity/clickhouse-backup/releases/download/v2.x.y/clickhouse-backup-linux-amd64.tar.gz \
     | sudo tar -xz -C /usr/local/bin/ clickhouse-backup
   ```
4. Configure rclone (one time, interactive: `rclone config`) creating remote `b2:` of type "Backblaze B2" using the keys above. Confirm with `rclone lsd b2:thebullpen-backups`.
5. Author `/etc/clickhouse-backup/config.yml`:
   ```yaml
   general:
     remote_storage: none
     backups_to_keep_local: 7
   clickhouse:
     username: bullpen
     password: ${CLICKHOUSE_PASSWORD}
     host: 127.0.0.1
     port: 9000
     data_path: /var/lib/thebullpen/clickhouse/data
     backups_path: /var/lib/thebullpen/backups
   ```
6. Author `ops/backup.sh`:
   ```bash
   #!/usr/bin/env bash
   set -euo pipefail
   source /etc/thebullpen/secrets.env

   STAMP=$(date -u +%Y%m%d-%H%M%SZ)
   BACKUP_NAME="bullpen-${STAMP}"

   # 1. ClickHouse snapshot
   clickhouse-backup create "$BACKUP_NAME"

   # 2. Checkpoint SQLite WAL, then copy
   sqlite3 /var/lib/thebullpen/registry.sqlite "PRAGMA wal_checkpoint(TRUNCATE);"
   cp /var/lib/thebullpen/registry.sqlite \
      "/var/lib/thebullpen/backups/${BACKUP_NAME}/registry.sqlite"

   # 3. Push to B2
   rclone copy --quiet "/var/lib/thebullpen/backups/${BACKUP_NAME}" \
       "b2:${B2_BUCKET_NAME}/daily/${BACKUP_NAME}"

   # 4. Weekly + monthly aliases
   if [[ $(date -u +%u) -eq 7 ]]; then
     rclone copyto "/var/lib/thebullpen/backups/${BACKUP_NAME}" \
         "b2:${B2_BUCKET_NAME}/weekly/${BACKUP_NAME}"
   fi
   if [[ $(date -u +%d) -eq 01 ]]; then
     rclone copyto "/var/lib/thebullpen/backups/${BACKUP_NAME}" \
         "b2:${B2_BUCKET_NAME}/monthly/${BACKUP_NAME}"
   fi

   # 5. Models artifact directory (weekly)
   if [[ $(date -u +%u) -eq 7 ]]; then
     rclone sync /var/lib/thebullpen/models \
         "b2:${B2_BUCKET_NAME}/models-snapshot/${STAMP}"
   fi

   # 6. Prune
   ops/scripts/prune-b2.sh
   ```
7. Author `ops/scripts/prune-b2.sh`: enforce retention 7-4-12 by listing each prefix in B2, sorted desc, keeping N newest, deleting older. (Plain `rclone delete` calls.)
8. Author the systemd unit + timer:
   ```ini
   # thebullpen-backup.service
   [Service]
   Type=oneshot
   User=thebullpen
   EnvironmentFile=/etc/thebullpen/secrets.env
   ExecStart=/home/%u/thebullpen/ops/backup.sh
   ```
   ```ini
   # thebullpen-backup.timer
   [Timer]
   OnCalendar=*-*-* 05:00:00
   Persistent=true
   Timezone=America/New_York
   [Install]
   WantedBy=timers.target
   ```
9. `sudo systemctl daemon-reload && sudo systemctl enable --now thebullpen-backup.timer`. Verify with `systemctl list-timers`.
10. Trigger manually: `sudo systemctl start thebullpen-backup.service`. Wait. Verify B2 has `daily/bullpen-<stamp>/` populated.
11. **Restore drill** (the discipline gate):
    1. Spin a separate ClickHouse container on a different port (`docker run --rm -p 18123:8123 -p 19000:9000 clickhouse/clickhouse-server:24.3`).
    2. Run `ops/scripts/restore-drill.sh` which:
       - downloads yesterday's backup from B2 to a temp dir,
       - uses `clickhouse-backup restore_remote` (or manual `restore` after `clickhouse-backup download`) into the temp instance,
       - copies the SQLite file into a temp location,
       - runs `SELECT count(*) FROM raw_statcast`, `pitches`, `prediction_log` against the temp CH and `SELECT * FROM bullpen_meta` against the temp SQLite,
       - asserts non-zero results (or zero if pre-load — adjust threshold per phase).
    3. Capture the output to `ops/runbooks/restore-drill-2026-XX-XX.log`.
12. Document `ops/runbooks/restore-drill.md` with the **exact** commands and expected output. (Recovery is on you in an emergency — your future self isn't going to remember.)
13. Add Grafana panel: time-since-last-successful-backup using a small text-file exporter (`echo $(date -u +%s) > /var/lib/thebullpen/last-backup-timestamp` at end of `backup.sh`, and a node_exporter textfile collector).

## Known edge cases

- **B2 bucket lifecycle vs. our pruner**: don't enable B2-side lifecycle rules — they'd race with our pruner. Pruner is authoritative.
- **clickhouse-backup permissions**: it needs to read `/var/lib/clickhouse/data` which is uid 101 inside the container. Run clickhouse-backup as a user that can `sudo` or run it inside the container via `docker exec`. Pick one and document.
- **SQLite WAL not checkpointed before copy**: backup looks correct but is missing transactions in the WAL. Always `PRAGMA wal_checkpoint(TRUNCATE)` first.
- **Bandwidth at deploy time**: a multi-GB backup uploads slowly. Don't run `deploy.sh` during the backup window.
- **B2 path conflicts**: weekly/monthly aliases are *copies*, not moves. If you also include them in the daily prefix, you'll double-pay storage. Pruner handles this — but the prune logic must be tested.
