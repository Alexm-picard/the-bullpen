## Scope boundaries

**IN scope**:
- Docker Engine installed inside WSL2
- ClickHouse single-node container managed by a systemd unit (Docker Compose under systemd)
- Persistent volumes on the host filesystem for data, config, logs
- Default user replaced with a dedicated `bullpen` ClickHouse user; password stored in `secrets.env`
- `clickhouse-jdbc` dependency added to backend `pom.xml`
- Spring connects on startup; one smoke INSERT/SELECT round-trip via integration test
- Flyway directory `backend/src/main/resources/db/migration/clickhouse/` created with `V001__baseline.sql` (no objects yet вЂ” schema lands per-leaf as needed)

**OUT of scope**:
- `pitches` / `pitches_live` / `prediction_log` schemas вЂ” those land in 1.2 / 3b.5
- Backups (clickhouse-backup) в†’ `0.10`
- TLS to ClickHouse вЂ” local-only access, plain TCP

## Objectives

1. ClickHouse 24.x runs under systemd, persists across `wsl --shutdown`.
2. Spring connects with `bullpen` user, password from `secrets.env`.
3. Flyway migration runner is wired and idempotent against ClickHouse.
4. Smoke test: Spring writes a row to a temp table, reads it back, deletes it.
5. Disk usage tracked: container's data volume reachable at `/var/lib/thebullpen/clickhouse/data/`.

## Step-by-step implementation tasks

1. Install Docker inside WSL (Docker Engine via apt, not Docker Desktop):
   ```bash
   sudo apt-get install -y docker.io docker-compose-plugin
   sudo usermod -aG docker thebullpen
   sudo systemctl enable --now docker
   ```
2. Author `infra/docker/docker-compose.clickhouse.yml`:
   ```yaml
   services:
     clickhouse:
       image: clickhouse/clickhouse-server:24.3
       container_name: bullpen-clickhouse
       restart: unless-stopped
       ulimits:
         nofile: { soft: 262144, hard: 262144 }
       volumes:
         - /var/lib/thebullpen/clickhouse/data:/var/lib/clickhouse
         - /var/lib/thebullpen/clickhouse/logs:/var/log/clickhouse-server
         - ../clickhouse/config.d:/etc/clickhouse-server/config.d:ro
         - ../clickhouse/users.d:/etc/clickhouse-server/users.d:ro
       environment:
         CLICKHOUSE_USER: bullpen
         CLICKHOUSE_PASSWORD_FILE: /run/secrets/ch_password
         CLICKHOUSE_DEFAULT_ACCESS_MANAGEMENT: 1
       secrets:
         - ch_password
       ports:
         - "127.0.0.1:8123:8123"
         - "127.0.0.1:9000:9000"
       mem_limit: 8g
   secrets:
     ch_password:
       file: /etc/thebullpen/clickhouse_password
   ```
3. Add `CLICKHOUSE_PASSWORD=<gen>` to `/etc/thebullpen/secrets.env`. Also write the same value to `/etc/thebullpen/clickhouse_password` (root:root, 0600) so the Docker secret can read it.
4. Author the systemd wrapper `clickhouse.service` that calls `docker compose -f infra/docker/docker-compose.clickhouse.yml up`. Standard restart policy.
5. `sudo systemctl daemon-reload && sudo systemctl enable --now clickhouse`.
6. Verify HTTP interface: `curl -u bullpen:<pw> http://127.0.0.1:8123/?query=SELECT+version()` returns `24.x.y`.
7. Add to `backend/pom.xml`:
   ```xml
   <dependency>
     <groupId>com.clickhouse</groupId>
     <artifactId>clickhouse-jdbc</artifactId>
     <version>0.6.x</version>
     <classifier>shaded</classifier>
   </dependency>
   <dependency>
     <groupId>org.flywaydb</groupId>
     <artifactId>flyway-core</artifactId>
   </dependency>
   ```
   Note: Flyway doesn't ship a ClickHouse module officially; use a thin custom migration runner OR `flyway-database-clickhouse` community plugin. Decision: custom migration runner вЂ” see step 9.
8. Add to `application.yml`:
   ```yaml
   bullpen:
     clickhouse:
       url: jdbc:clickhouse://127.0.0.1:8123/default
       user: bullpen
       password: ${CLICKHOUSE_PASSWORD}
   ```
9. Author `ClickhouseDataSourceConfig` exposing a `DataSource` bean using `clickhouse-jdbc`. Author a small `ClickhouseMigrationRunner` (`@Component`) that on startup reads `db/migration/clickhouse/V*.sql` files in lexical order and applies any not yet recorded in a `_schema_migrations` ClickHouse table (custom вЂ” Flyway doesn't natively support ClickHouse). Idempotent.
10. Author `ClickhouseSmokeRepository`:
    ```java
    public List<Long> roundTrip() {
        jdbcTemplate.execute("CREATE TEMPORARY TABLE smoke (n UInt64)");
        jdbcTemplate.update("INSERT INTO smoke VALUES (1),(2),(3)");
        return jdbcTemplate.queryForList("SELECT n FROM smoke ORDER BY n", Long.class);
    }
    ```
11. Author `ClickhouseSmokeIT` (integration test) using Testcontainers ClickHouse to assert the smoke repository works against a real container.
12. Run `./mvnw verify` вЂ” Testcontainers spins up ephemeral ClickHouse, smoke test passes.
13. Restart `thebullpen-api`. Confirm logs show `Migration runner: applied V001__baseline.sql` (or `nothing to apply` on second start).

## Known edge cases

- **`mem_limit` ignored**: cgroup v2 in WSL2 may reject `mem_limit`. Use `deploy.resources.limits.memory` form for compose v3 if needed.
- **Container DNS**: ClickHouse must bind to `127.0.0.1`, not `0.0.0.0`. WSL2's port-forwarding will expose to Windows host but not the public LAN.
- **Permissions on data volume**: ClickHouse runs as uid 101 inside the container. Either `chown 101:101 /var/lib/thebullpen/clickhouse/data` or `--user $(id -u):$(id -g)` and bind-mount.
- **Flyway-vs-custom**: a community ClickHouse Flyway plugin exists but versioning is shaky. Custom migration runner is ~50 lines of code; pinned in this leaf.
- **`secrets.env` double-quoting**: Docker Compose `secrets.env` and systemd `EnvironmentFile=` parse differently вЂ” keep the file plain `KEY=value` with no quotes.
