# 0.9 — External monitoring (Uptime Robot + Healthchecks.io + Discord)

> Owning phase: `phase-0-foundation` · Estimated effort: `2–3 h` · Authored: 2026-05-10
> · Revised: 2026-05-23 — external uptime vendor switched from Better Stack to
> Uptime Robot per decision [129]. Substance of the plan is unchanged
> (same single-region HTTP probe, same Discord webhook integration, same
> two-failure debounce). What differs: probe interval shifts from 30s
> (Better Stack free tier) to 5 min (Uptime Robot free tier; paid tiers go
> to 1 min); the Discord alert channel is wired via Uptime Robot's webhook
> alert contact rather than Better Stack's native Discord integration.

## Scope boundaries

**IN scope**:

- Uptime Robot HTTP(S) monitor configured against `https://api.thebullpen.net/v1/actuator/health` every 5 min
- Healthchecks.io account + a single placeholder check for the worker (heartbeat URL stored in `secrets.env`)
- Discord webhook URL stored in `secrets.env`
- A small Spring `@Component` (`DiscordAlerter`) that sends formatted webhook messages
- A CLI script (`ops/scripts/test-discord-alert.sh`) that fires a test message
- Alert thresholds documented in `00-OBSERVABILITY-STRATEGY.md` reviewed and reconciled

**OUT of scope**:

- Real drift / calibration alerts → Phase 3c
- Real heartbeat firing from worker `@Scheduled` jobs → those jobs land in Phase 3c/3d
- Healthchecks.io coverage of every cron — only one placeholder for now

---

## Objectives

1. Uptime Robot pages within ~6 minutes of API going down (one missed check + one confirmation on the 5 min interval).
2. Discord webhook receives a test message with the agreed template.
3. `secrets.env` carries `DISCORD_WEBHOOK_URL`, `HEALTHCHECKS_IO_PING_URL`.
4. A future leaf can `@Autowired DiscordAlerter` and `.page("title", details)` without re-discovering this code.

## Step-by-step implementation tasks

1. Create accounts (free tier) on Uptime Robot and Healthchecks.io.
2. In Uptime Robot, create an HTTP(S) monitor on `https://api.thebullpen.net/v1/actuator/health`, 5 min interval, "Down after 2 failed attempts". Add an Alert Contact of type `Webhook` pointing at the Discord webhook URL (use Discord's `?wait=true` query suffix on the URL so Uptime Robot's POST body is delivered as a Discord message — Uptime Robot's webhook payload is JSON-compatible with Discord's `content` field via the "Send as JSON" toggle + a small payload template).
3. In Healthchecks.io, create a placeholder check `worker-heartbeat-placeholder` (real schedule wired up in Phase 3+). Copy the ping URL.
4. Create a Discord server (or reuse existing). Create channel `#bullpen-alerts`. Add a webhook; copy URL.
5. Append to `/etc/thebullpen/secrets.env`:
   ```
   DISCORD_WEBHOOK_URL=https://discord.com/api/webhooks/...
   HEALTHCHECKS_IO_PING_URL=https://hc-ping.com/...
   ```
   (Uptime Robot's HTTP probe doesn't need a secret on our side; outbound only.)
6. `sudo systemctl restart thebullpen-api thebullpen-worker` to pick up new env.
7. Author `AlertSeverity` enum.
8. Author `DiscordAlerter`:
   ```java
   @Component
   public class DiscordAlerter {
     private final RestClient http;
     private final String webhookUrl;
     public DiscordAlerter(@Value("${bullpen.discord.webhook-url}") String webhookUrl) {
       this.webhookUrl = webhookUrl;
       this.http = RestClient.create();
     }
     public void send(AlertSeverity severity, String title, Map<String,String> fields) {
       String prefix = severity == AlertSeverity.PAGE ? "🔴 PAGE" : "🟡 NOTICE";
       String body = formatMarkdown(prefix, title, fields);
       http.post().uri(webhookUrl).contentType(MediaType.APPLICATION_JSON)
           .body(Map.of("content", body)).retrieve().toBodilessEntity();
     }
     // ... formatMarkdown helper following 00-OBSERVABILITY-STRATEGY.md template
   }
   ```
9. Author `DiscordAlerterTest` using `MockRestServiceServer` or WireMock. Assert payload structure.
10. Author `ops/scripts/test-discord-alert.sh`:
    ```bash
    #!/usr/bin/env bash
    set -euo pipefail
    source /etc/thebullpen/secrets.env
    curl -fsS -H 'Content-Type: application/json' \
      -d '{"content":"🟡 NOTICE — manual smoke test from 0.9"}' \
      "$DISCORD_WEBHOOK_URL"
    echo " ✓ delivered"
    ```
    `chmod +x` it. Run it. Confirm the message appears in `#bullpen-alerts`.
11. End-to-end pager test: `sudo systemctl stop thebullpen-api`. Wait one full check interval (5 min) plus the confirmation check (~6 min total). Uptime Robot should page, posting to Discord. Restart with `systemctl start`.
12. Document the runbook at `ops/runbooks/external-monitoring.md` covering: how to add a new Uptime Robot monitor, how to wire a real Healthchecks.io check from a `@Scheduled` Spring job, how to rotate the Discord webhook.

## Known edge cases

- **Webhook rate limits**: Discord limits per webhook ~30 req/min. Real alert volume is far below. If the drift detector ever bursts (Phase 3c), batch in DiscordAlerter (queue + 1/sec flusher).
- **Webhook URL leak**: rotating it requires regenerating in Discord, updating `secrets.env`, restarting Spring. Document this in runbook.
- **Uptime Robot single-region probe**: free tier probes from one US region. A single-region false-positive is debounced by the 2-failure rule (decision baked above). If we ever observe regional flapping, upgrade to multi-region (paid) rather than lower the debounce.
- **Detection latency floor**: 5 min interval + 2-failure debounce = ~6 min worst-case time-to-page. Acceptable for portfolio scale; if tightening becomes important, the upgrade path is Uptime Robot Pro (1 min interval) — one billing change, no code change.
- **Local-only WSL during outages**: external monitor tells you when the host went down; internal Prometheus can't (decision [17]).
