# 0.3 — Spring skeleton (api + worker profiles)

> Owning phase: `phase-0-foundation` · Estimated effort: `6–8 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Maven-based Spring Boot 3.x project under `backend/`
- Java 21, virtual threads enabled
- Two Spring profiles (`api`, `worker`) wired to two systemd units (`thebullpen-api.service`, `thebullpen-worker.service`)
- `/v1/health` and `/actuator/health` endpoints reachable per profile
- `springdoc-openapi` exposing `/v3/api-docs` and Swagger UI in `api` profile
- Bean Validation (`jakarta.validation`) wired
- `MockMvc` test scaffold present
- API base prefix `/v1/` baked in via `server.servlet.context-path`
- Logback `JsonEncoder` configured to write structured logs to stdout (journald-ready)
- Empty `data/`, `domain/`, `inference/`, `registry/`, `drift/`, `retraining/`, `ingest/`, `simulation/`, `config/` packages

**OUT of scope**:
- ClickHouse / SQLite connections → `0.4`, `0.5`
- Real prediction logic → Phase 1
- Auth on `/admin/*` → Phase 3a (when admin endpoints first exist)

---

## Objectives

1. Project builds reproducibly with `./mvnw package`.
2. Both systemd units start cleanly and listen on configured ports.
3. `/v1/health` returns 200 from the `api` unit; `/actuator/health` returns 200 on both.
4. `/v3/api-docs` returns a valid OpenAPI 3.x JSON document on the `api` unit.
5. Virtual threads visible: a request to a sleeping endpoint runs on `VirtualThread`.
6. JSON logging works: `journalctl -u thebullpen-api -f` shows one-line JSON events.

## Step-by-step implementation tasks

1. Generate Spring Boot project (Spring Initializr CLI or manual `pom.xml`):
   - `spring-boot-starter-web`, `spring-boot-starter-actuator`, `spring-boot-starter-validation`, `spring-boot-starter-security`, `springdoc-openapi-starter-webmvc-ui`, `logstash-logback-encoder`.
2. Set Java 21 in `pom.xml`. Add `<spring-boot.version>3.x.y</spring-boot.version>` matching the latest 3.x.
3. Author `BullpenApplication` (single `main`).
4. Author `application.yml`:
   ```yaml
   spring:
     application:
       name: thebullpen
     threads:
       virtual:
         enabled: true
   server:
     servlet:
       context-path: /v1
   management:
     endpoints:
       web:
         exposure:
           include: health,info,prometheus
     endpoint:
       health:
         probes:
           enabled: true
         group:
           readiness:
             include: readinessState,db
   ```
5. Author `application-api.yml` (port 8080) and `application-worker.yml` (port 8081, plus `management.server.port: 8081`).
6. Author `HealthController`:
   ```java
   @RestController
   @Profile("api")
   public class HealthController {
       @GetMapping("/health")
       public Map<String, Object> health() {
           return Map.of("status", "ok", "thread", Thread.currentThread().toString());
       }
   }
   ```
7. Author placeholder `SecurityConfig` permitting all for now (real basic-auth for `/admin/*` lands in Phase 3a):
   ```java
   @Configuration
   public class SecurityConfig {
       @Bean
       SecurityFilterChain chain(HttpSecurity http) throws Exception {
           return http
               .csrf(c -> c.disable())
               .authorizeHttpRequests(a -> a.anyRequest().permitAll())
               .build();
       }
   }
   ```
8. Author `OpenApiConfig` setting `info.title=The Bullpen API`, `info.version=v1`, base URL `/v1`.
9. Author `logback-spring.xml` using `LogstashEncoder`. Mask sensitive keys (`password`, `Authorization`, `webhook_url`, `api_key`, `B2_*`, `CLICKHOUSE_PASSWORD`).
10. Author `HealthControllerTest` using `@WebMvcTest(HealthController.class)` + `@ActiveProfiles("api")`.
11. Author systemd units (see `00-DEPLOYMENT-STRATEGY.md` shape). Differ only in `--spring.profiles.active=api|worker` and `MemoryMax` (4G / 2G).
12. Build: `./mvnw -DskipTests=false package`. Install JAR: `sudo install -o thebullpen -m 0644 backend/target/*.jar /opt/thebullpen/app.jar`.
13. Reload + start: `sudo systemctl daemon-reload && sudo systemctl enable --now thebullpen-api thebullpen-worker`.
14. Verify:
    ```bash
    curl http://localhost:8080/v1/health
    curl http://localhost:8080/v1/actuator/health
    curl http://localhost:8081/v1/actuator/health
    curl http://localhost:8080/v1/v3/api-docs | jq .openapi
    ```
15. Verify virtual threads: hit `/v1/health` and read the JSON `thread` field — must contain `VirtualThread`.

## Known edge cases

- **Profile bleeding**: omitting `@Profile` on a controller makes it visible in both units. Worker unit must NOT serve `/v1/predict/*` — enforce via `@Profile("api")` on every controller.
- **Actuator on worker**: management port `8081` doesn't conflict with the API. Test that `/actuator/prometheus` is reachable on both for Prometheus scraping (0.8).
- **Virtual-thread compatibility**: avoid `synchronized` blocks during long I/O — they pin to platform threads. Document in `00-CONVENTIONS.md` if discovered.
- **Logback `LogstashEncoder` masking**: must run before line buffering; verify with `journalctl` against a deliberately-leaky log statement.
