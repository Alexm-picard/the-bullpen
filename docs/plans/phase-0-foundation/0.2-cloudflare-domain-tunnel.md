# 0.2 — Cloudflare domain + Tunnel

> Owning phase: `phase-0-foundation` · Estimated effort: `3–4 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Domain `thebullpen.net` registered (or transferred) and pointed at Cloudflare nameservers
- DNS records:
  - `thebullpen.net` → CNAME to Vercel (created in 0.6, placeholder for now)
  - `api.thebullpen.net` → CNAME to Cloudflare Tunnel target
- `cloudflared` installed inside WSL2, managed by systemd, running as the `thebullpen` user
- Tunnel created in Cloudflare Zero Trust dashboard, token stored in `/etc/thebullpen/secrets.env`
- Tunnel proves end-to-end with a hello-world HTTP listener (8080) returning 200 from the public internet via `curl https://api.thebullpen.net/hello`

**OUT of scope**:
- Spring app behind the tunnel → that's `0.3-spring-skeleton.md` (this leaf uses a placeholder responder)
- TLS certificate management — Cloudflare handles it edge-side; nothing to install

---

## Objectives

1. Apex `thebullpen.net` resolves through Cloudflare DNS.
2. `api.thebullpen.net` reachable from the public internet, terminating at the local hello responder via Cloudflare Tunnel.
3. `cloudflared` runs as a managed systemd unit with restart policy.
4. Tunnel token stored in `/etc/thebullpen/secrets.env`, never in git.
5. Public IP is **not** exposed; no port forwarding on the local router.

## Step-by-step implementation tasks

1. Register `thebullpen.net` (Cloudflare Registrar preferred, or transfer to it). Wait for nameservers to propagate.
2. In Cloudflare dashboard → Zero Trust → Networks → Tunnels → create tunnel `thebullpen-prod`. Copy the connector token.
3. Add token to `/etc/thebullpen/secrets.env`:
   ```bash
   sudo install -o root -g root -m 0600 /dev/null /etc/thebullpen/secrets.env
   sudo tee -a /etc/thebullpen/secrets.env >/dev/null <<EOF
   CLOUDFLARED_TUNNEL_TOKEN=<paste-token>
   EOF
   ```
4. Install `cloudflared` inside WSL:
   ```bash
   curl -L https://pkg.cloudflare.com/cloudflare-main.gpg | sudo tee /usr/share/keyrings/cloudflare-main.gpg >/dev/null
   echo 'deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared noble main' \
     | sudo tee /etc/apt/sources.list.d/cloudflared.list
   sudo apt-get update && sudo apt-get install -y cloudflared
   ```
5. Create `/etc/systemd/system/cloudflared.service`:
   ```ini
   [Unit]
   Description=Cloudflare Tunnel
   After=network-online.target
   Wants=network-online.target

   [Service]
   Type=simple
   User=thebullpen
   Group=thebullpen
   EnvironmentFile=/etc/thebullpen/secrets.env
   ExecStart=/usr/bin/cloudflared tunnel --no-autoupdate run --token ${CLOUDFLARED_TUNNEL_TOKEN}
   Restart=on-failure
   RestartSec=10s
   StartLimitBurst=5
   StartLimitIntervalSec=300s
   StandardOutput=journal
   StandardError=journal

   [Install]
   WantedBy=multi-user.target
   ```
6. `sudo systemctl daemon-reload && sudo systemctl enable --now cloudflared`. Confirm `systemctl status cloudflared` is green.
7. Stand up a hello-world responder on `:8080` (one-liner):
   ```bash
   python3 -m http.server 8080
   # OR a netcat one-shot returning 200
   ```
8. In Cloudflare dashboard → tunnel → Public Hostnames: add `api.thebullpen.net` → service `http://localhost:8080`.
9. From any public network: `curl -i https://api.thebullpen.net/`. Must return 200. TLS is Cloudflare's edge cert.
10. Commit `infra/systemd/cloudflared.service` and `infra/cloudflared/config.yml` (no secrets) to git. Document edge config in `ops/runbooks/cloudflared-recovery.md`.

## Known edge cases

- **Token rotation**: tokens can be invalidated from the dashboard. Runbook must say: invalidate → create new connector → replace value in `secrets.env` → `systemctl restart cloudflared`.
- **WSL DNS regression**: `generateResolvConf=true` in `/etc/wsl.conf` (set in 0.1) prevents Windows DNS edits from breaking egress.
- **IPv6 weirdness**: Cloudflare prefers IPv6 if available. If you see odd routing, try `cloudflared tunnel run --protocol=http2`.
