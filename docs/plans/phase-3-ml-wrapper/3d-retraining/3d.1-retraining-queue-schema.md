# 3d.1 — `retraining_queue` service layer

> Owning phase: `phase-3-ml-wrapper/3d-retraining` · Estimated effort: `3–4 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:

- Service layer over `retraining_queue` (table from 3a.1)
- `RetrainingQueueService.enqueue(modelName, triggerType, metadata)` — returns `triggerId` (UUID)
- Idempotency via the `trigger_id UNIQUE` constraint
- `claimNext()` — atomically transitions queued → running for the worker to pick up
- `complete(triggerId, succeeded, producedVersionId, errorMessage)`
- Status transitions: `queued → running → succeeded|failed`; `cancelled` is terminal too

**OUT of scope**:

- The actual triggers → `3d.2`
- Python retrain job → `3d.3`
- systemd timer → `3d.4`

---

## Objectives

1. Service supports enqueue / claim / complete / cancel.
2. Concurrent claims are safe (only one worker gets the row).
3. Trigger-id uniqueness prevents duplicate enqueues.

## Step-by-step implementation tasks

1. Author DTOs and enums.
2. Author `RetrainingQueueRepository`:
   ```java
   public RetrainingTrigger insert(RetrainingTrigger t) { ... }   // throws DuplicateKeyException on duplicate trigger_id
   public Optional<RetrainingTrigger> findByTriggerId(String triggerId) { ... }
   public Optional<RetrainingTrigger> claimNextQueued() {
       // BEGIN TRANSACTION;
       // SELECT * FROM retraining_queue WHERE status='queued' ORDER BY enqueued_at LIMIT 1;
       // UPDATE ... SET status='running', started_at=now() WHERE id=? AND status='queued';
       // COMMIT;
       // Return updated row only if UPDATE affected 1 row.
   }
   public void markComplete(String triggerId, boolean ok, Long versionId, String error) { ... }
   public void cancel(String triggerId) { ... }
   ```
3. Author `RetrainingQueueService`:
   - `enqueue(modelName, triggerType, metadata)` constructs `triggerId = UUID.randomUUID()` and inserts. Returns trigger id.
   - `claim()` for the worker.
   - `complete(...)` and `cancel(...)`.
4. Author `RetrainingQueueServiceIT`:
   - Enqueue, claim (succeeds), claim again (returns empty).
   - Concurrent test: 10 threads call `claim()`; exactly one succeeds, others empty.
   - Duplicate trigger_id throws (defensive sanity).
   - Mark complete OK / FAIL paths.

## Known edge cases

- **Long-running job crashed mid-flight**: row stays in `running` forever. Add a "stale-claim reaper" that flips `running` rows older than 4 hours back to `queued` (or to `failed` with `error_message='claim_timeout'`). Implement now or as a 3d.4 follow-up.
- **`produced_version_id` references a deleted model_version**: shouldn't happen (we never delete). Foreign key is enforced.
- **Cancel during running**: allowed; the Python job must check status periodically and abort.
