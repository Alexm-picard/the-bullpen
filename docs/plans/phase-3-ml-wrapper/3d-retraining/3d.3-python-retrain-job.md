# 3d.3 — Python retrain job

> Owning phase: `phase-3-ml-wrapper/3d-retraining` · Estimated effort: `8–10 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:

- A single Python entry point `python -m thebullpen.retraining.run` that:
  1. Claims the next queued row via Spring's `claimNext` (over a thin REST API)
  2. Branches on `model_name` to run the matching training pipeline (2a / 2b / 2c)
  3. Calls Spring's registry `register` API on success
  4. Calls Spring's `markComplete` on success or failure
  5. Propagates `trigger_id` into all logs and `metadata.json`
- Atomic registration: partial failures don't leave half-trained models
- Hyperparameters fixed within the run (decision `[81]`); HP search is v1.5

**OUT of scope**:

- The training pipelines themselves — already exist (Phase 2)
- systemd timer → 3d.4
- Auto-promotion → never (decision `[44]`); job stops at `register` → stage=`candidate`.

## Step-by-step implementation tasks

1. Author `_api_client.py`:
   ```python
   class BullpenAdminClient:
       def __init__(self, base_url, auth: tuple[str,str]): ...
       def claim_next_queued(self) -> dict | None: ...
       def register(self, model_name: str, version: str, artifact_dir: Path, eval_metrics: dict) -> int: ...
       def mark_complete(self, trigger_id: str, succeeded: bool, version_id: int|None, error: str|None) -> None: ...
   ```
2. Author `_dispatch.py`:
   ```python
   DISPATCH = {
     "pitch_outcome_pre": train_pitch_pre,
     "pitch_outcome_post": train_pitch_post,
     "pitch_outcome_lr_baseline": train_lr_baseline,
     "batted_ball": train_batted_ball_mlp,
     "batted_ball_lgbm_baseline": train_batted_ball_lgbm,
   }
   ```
3. Author `run.py`:
   ```python
   def main():
       client = BullpenAdminClient(...)
       trig = client.claim_next_queued()
       if not trig: return
       trigger_id = trig["trigger_id"]
       structlog.contextvars.bind(trigger_id=trigger_id, model_name=trig["model_name"])
       try:
         next_version = next_version_label(trig["model_name"])
         artifact_dir = DISPATCH[trig["model_name"]](trigger_id=trigger_id, version=next_version, ...)
         eval_metrics = json.loads((artifact_dir / "eval" / "metrics.json").read_text())
         vid = client.register(trig["model_name"], next_version, artifact_dir, eval_metrics)
         client.mark_complete(trigger_id, succeeded=True, version_id=vid, error=None)
       except Exception as e:
         log.exception("Retrain failed")
         client.mark_complete(trigger_id, succeeded=False, version_id=None, error=str(e))
         raise
   ```
4. Each training function (`train_pitch_pre`, etc.) must accept `trigger_id` and persist it into `metadata.json`.
5. Author `test_run.py`:
   - Mock `BullpenAdminClient` returning a queued trigger; mock training function returning a path; verify `register` and `mark_complete(succeeded=True)` called.
   - Mock training raising an exception; verify `mark_complete(succeeded=False)` called.
6. End-to-end test (manual; not in CI): enqueue via admin endpoint, run `python -m thebullpen.retraining.run`, verify new candidate appears in registry.

## Known edge cases

- **Two parallel claims**: the `claim_next_queued` API returns one to the first caller; second call gets empty. The job is single-instance per host (timer fires hourly).
- **Crash mid-train**: row stuck in `running`. Stale-claim reaper from 3d.1 recovers.
- **`next_version_label` collision**: if version `v17` exists and you try to register again as `v17`, registry rejects (UNIQUE constraint). The job auto-increments to `v18`.
- **Atomic registration**: artifact files placed first, then DB row inserted in 3a.5's flow. Failure partway → cleanup script.
- **Trigger metadata propagation**: `trigger_metadata` JSON gets copied into `metadata.json` so future you can correlate "this v17 came from drift trigger of 2026-06-15".
