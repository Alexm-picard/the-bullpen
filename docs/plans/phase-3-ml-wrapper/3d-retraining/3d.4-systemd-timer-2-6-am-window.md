# 3d.4 — systemd timer (2–6 AM ET window)

> Owning phase: `phase-3-ml-wrapper/3d-retraining` · Estimated effort: `2–3 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- systemd timer firing the Python retrain job hourly between 2–6 AM ET (decision `[19]` GPU scheduling)
- The timer is host-side; runs `python -m thebullpen.retraining.run` once per fire
- Stale-claim reaper: a separate timer that flips orphaned `running` rows back to `queued` (or `failed`) every 30 minutes
- `infra/systemd/thebullpen-retrain.service` + `.timer`
- `infra/systemd/thebullpen-stale-claim-reaper.service` + `.timer`

**OUT of scope**:
- Job queue with GPU lock — v1.5 (decision `[19]` Option B)
- Concurrent retrains — single instance per host

---

## Objectives

1. Retrain timer fires hourly within the 2–6 AM ET window.
2. Each fire calls the Python job, which claims one row from the queue (or no-op if empty).
3. Self-healing: a job that collides with another GPU workload fails; drift detector re-triggers next cycle.
4. Stale-claim reaper recovers stuck rows.

## Step-by-step implementation tasks

1. Author `thebullpen-retrain.service`:
   ```ini
   [Unit]
   Description=The Bullpen — Retraining Job
   After=network-online.target thebullpen-api.service

   [Service]
   Type=oneshot
   User=thebullpen
   Group=thebullpen
   WorkingDirectory=/home/thebullpen/thebullpen/ml
   EnvironmentFile=/etc/thebullpen/secrets.env
   ExecStart=/usr/local/bin/uv run python -m thebullpen.retraining.run
   StandardOutput=journal
   StandardError=journal
   ```
2. Author `thebullpen-retrain.timer`:
   ```ini
   [Timer]
   OnCalendar=*-*-* 02..06:00:00 America/New_York
   Persistent=false
   AccuracySec=1m
   [Install]
   WantedBy=timers.target
   ```
3. Author the stale-claim-reaper service + timer:
   ```ini
   # service
   [Service]
   Type=oneshot
   User=thebullpen
   ExecStart=/usr/bin/curl -fsS -u admin:$(...) -X POST http://localhost:8080/v1/admin/retrain/reap-stale
   ```
   ```ini
   # timer
   [Timer]
   OnCalendar=*-*-* *:00,30:00
   ```
   And add a thin admin endpoint `POST /v1/admin/retrain/reap-stale` that calls a service method `RetrainingQueueService.reapStaleClaims(Duration.ofHours(4))`.
4. `sudo systemctl daemon-reload && sudo systemctl enable --now thebullpen-retrain.timer thebullpen-stale-claim-reaper.timer`.
5. Verify `systemctl list-timers` shows both timers with next-run times in the expected windows.
6. Document recovery in `ops/runbooks/retraining-failure-recovery.md`: what to do when a job fails, how to manually re-enqueue, how to inspect logs.
7. End-to-end test (manual):
   - Enqueue via admin endpoint.
   - Wait for next hourly fire (or trigger manually with `systemctl start thebullpen-retrain.service`).
   - Verify new candidate registered.
   - Verify Healthchecks.io heartbeat fired.

## Known edge cases

- **Daylight saving time**: systemd `OnCalendar=...:00 America/New_York` handles DST correctly; verify after the spring forward / fall back transitions.
- **Job collision with gaming / other GPU work**: PyTorch will raise `CUDA out of memory`. Job catches and marks `failed`. DriftTrigger may re-enqueue next day.
- **Concurrent timer + manual API**: the queue service ensures only one row claimed at a time. Manual enqueue + timer next fire is safe.
- **Reaper aggression**: 4-hour stale threshold matches the longest expected training time (full batted-ball MLP retrain). If retraining grows past 4h, bump the threshold.
