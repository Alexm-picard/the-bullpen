# 3d.2 — Three triggers: scheduled / drift / manual

> Owning phase: `phase-3-ml-wrapper/3d-retraining` · Estimated effort: `4–6 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Three Spring components, each enqueuing into the queue:
  - `ScheduledTrigger`: monthly cron — enqueues all active model_names (decision `[79]` floor)
  - `DriftTrigger`: reads recent `drift_metrics` and enqueues if calibration > 1.5× training cal sustained 7 days (decision `[80]` — tighter than alert threshold)
  - `ManualTrigger`: admin endpoint `POST /v1/admin/retrain` — enqueues a single model
- All three share the `RetrainingQueueService` from 3d.1
- Hybrid floor + ceiling design (`[79]`)

**OUT of scope**:
- Python retrain job → 3d.3
- systemd timer for the runner → 3d.4

---

## Objectives

1. Monthly trigger enqueues active models on the 1st of each month (UTC midnight ≈ 7-8 PM ET prior day).
2. Drift trigger queries `drift_metrics` and only enqueues when conditions met.
3. Manual trigger reachable via admin endpoint, dedup'd by recent enqueue (don't double-enqueue if a job is already queued/running).

## Step-by-step implementation tasks

1. Author `ScheduledTrigger`:
   ```java
   @Component @Profile("worker")
   class ScheduledTrigger {
     @Scheduled(cron = "0 0 0 1 * *", zone = "America/New_York")
     public void monthly() {
       for (var model : registry.activeModelNames()) {
         queue.enqueue(model, TriggerType.SCHEDULED, Map.of("schedule","monthly"));
       }
     }
   }
   ```
2. Author `DriftTrigger`:
   ```java
   @Component @Profile("worker")
   class DriftTrigger {
     @Scheduled(cron = "0 0 4 * * *", zone = "America/New_York")
     public void daily() {
       for (var model : registry.activeModels()) {
         var calRecent = driftRepo.findLast7Days(model.name(), MetricType.CALIBRATION_ERROR, "all");
         double trainingCal = readTrainingCalibration(model);
         boolean sustainedDrift = calRecent.size() >= 7
           && calRecent.stream().allMatch(m -> m.value() > 1.5 * trainingCal);
         if (sustainedDrift && !queue.isAlreadyQueuedRecently(model.name(), Duration.ofDays(7))) {
           queue.enqueue(model.name(), TriggerType.DRIFT, Map.of("calibration_threshold", 1.5, "sustained_days", 7));
           discord.send(NOTICE, "Drift-triggered retrain enqueued for " + model.name(), ...);
         }
       }
     }
   }
   ```
3. Author `ManualTrigger` + `RetrainAdminController`:
   ```java
   @PostMapping("/v1/admin/retrain")
   public RetrainingTrigger manual(@RequestBody ManualRetrainRequest req) {
     return queue.enqueue(req.modelName(), TriggerType.MANUAL, req.metadata());
   }
   ```
4. Author `TriggersIT`:
   - Synthetic 7 days of drift > 1.5× cal → DriftTrigger enqueues exactly once.
   - Same condition next day → DriftTrigger does NOT re-enqueue (dedup).
   - Manual API enqueues; second call with same modelName within 1h dedups.
   - ScheduledTrigger enqueues all active models once per call.

## Known edge cases

- **Drift trigger threshold (1.5× / 7 days) tighter than alerting (1.5× / 3 days)** — by design, 4-day human investigation window (decision `[80]`).
- **Off-season cron**: November 1 fires; the model has no recent data to retrain on. The Python job (3d.3) handles this — it skips if no new data since last train.
- **Manual API auth**: `/v1/admin/retrain` requires HTTP basic (already covered by SecurityConfig from 3a.4).
