# 3a.2 — Registry service CRUD

> Owning phase: `phase-3-ml-wrapper/3a-registry` · Estimated effort: `5–7 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Spring service `RegistryService` exposing CRUD operations on `model_versions` and `model_routing`
- `register(modelVersion)` — atomic insert with stage='candidate'; verifies artifact paths exist on disk
- `getById(id)`, `findByName(modelName)`, `getActiveRouting(modelName)`
- `updateStage(id, newStage)` — internal helper used by 3a.4's promotion API
- Constructor-injected JdbcTemplate, no JPA
- Comprehensive integration tests against in-memory SQLite

**OUT of scope**:
- Feature-schema-hash enforcement at register time → `3a.3`
- HTTP endpoints (admin API) → `3a.4`
- Snapshot storage to B2 → `3a.5`

---

## Objectives

1. Service supports all CRUD ops needed by 3a.3, 3a.4, 3b.
2. `register` is idempotent on `(model_name, version)` — second call with the same key returns the existing row.
3. State transitions enforced by service layer (e.g., `archived` is terminal).

## Step-by-step implementation tasks

1. Author `Stage` enum: `CANDIDATE`, `SHADOW`, `CHAMPION`, `ARCHIVED`.
2. Author `ModelVersion` record matching the `model_versions` schema.
3. Author `RegistryRepository`:
   ```java
   public ModelVersion insert(ModelVersion mv) { ... }
   public Optional<ModelVersion> findById(long id) { ... }
   public List<ModelVersion> findByName(String modelName) { ... }
   public ModelVersion updateStage(long id, Stage newStage) { ... }
   public Optional<ModelVersion> findChampion(String modelName) { ... }
   public Optional<ModelVersion> findChallenger(String modelName) { ... }
   ```
4. Author `RegistryService` orchestrating:
   - `register(RegisterRequest)`:
     1. Validate that artifact files exist on disk (`model.onnx`, `metadata.json`, `feature_pipeline.json`, `training_data.parquet`).
     2. Check `(model_name, version)` doesn't already exist.
     3. Insert with stage=`CANDIDATE`.
   - `transitionStage(id, newStage)`:
     - Validates allowed transitions: `CANDIDATE → SHADOW`, `CANDIDATE → CHAMPION`, `SHADOW → CHAMPION`, any → `ARCHIVED`.
     - When promoting to `CHAMPION`: archive the previous champion atomically (single transaction).
5. Define `RegistryException` subclasses: `ArtifactMissing`, `DuplicateVersion`, `IllegalTransition`.
6. Author `RegistryServiceIT` (Testcontainers SQLite):
   - Register a model with valid artifacts → returns row.
   - Register again with same key → returns same row (idempotent).
   - Register with missing files → throws `ArtifactMissing`.
   - `transitionStage(CANDIDATE, ARCHIVED)` → ok.
   - `transitionStage(CANDIDATE, CANDIDATE)` → throws `IllegalTransition`.
   - Promoting to `CHAMPION` archives the previous champion (atomic).

## Known edge cases

- **No champion yet**: `findChampion` returns Optional.empty(). Promotion API (3a.4) must handle "first promotion ever".
- **Two champions**: should never happen. Add a DB-level constraint OR a service-level invariant test that runs at startup.
- **Stage transition during snapshot save**: out of scope here; 3a.5 handles.
- **`SHADOW → CANDIDATE` (demotion)**: not allowed. To "undo a shadow": archive it.
