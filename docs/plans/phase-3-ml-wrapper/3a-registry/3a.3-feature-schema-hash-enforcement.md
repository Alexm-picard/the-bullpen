# 3a.3 ‚Äî Feature schema hash enforcement

> Owning phase: `phase-3-ml-wrapper/3a-registry` ¬∑ Estimated effort: `4‚Äì6 h` ¬∑ Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- At `register()` time, compute the SHA-256 of the candidate's `feature_pipeline.json` (canonicalized) and verify it matches the production pipeline for that `model_name`
- Bootstrap rule: first registration for a given `model_name` sets the production pipeline; subsequent must match (Risk Register G3 resolution)
- Reset escape hatch: `registerWithBootstrap(modelName, version)` with explicit confirmation flag, archives all prior versions
- Java-side computation must match Python-side exactly (canonical JSON serialization)

**OUT of scope**:
- The pipeline contents themselves (Phase 2a.1/2a.2 already define these)
- Promotion API ‚Üí `3a.4`

**Hard rule**: this is the Discipline-7 enforcement. Without it, the registry is just a database.

---

## Objectives

1. Mismatched-hash registrations throw `FeatureSchemaMismatch` with a clear diff.
2. First-ever registration for a model_name succeeds (bootstrap path).
3. Java-side hash computation matches Python's exact byte sequence given the same JSON.
4. Reset escape hatch requires `--reset-feature-schema` AND archives all prior versions in a single transaction.

## Step-by-step implementation tasks

1. Author `CanonicalJson`: writes JSON with sorted keys, Unicode escaping minimized, no whitespace, exact float-formatting rules. Defines the canonical byte sequence for hashing.
2. Author `FeatureSchemaHasher`:
   ```java
   public String compute(Path featurePipelineJson) {
       String content = Files.readString(featurePipelineJson);
       Object parsed = mapper.readValue(content, Object.class);
       String canonical = canonicalJson.serialize(parsed);
       return "sha256:" + sha256Hex(canonical.getBytes(StandardCharsets.UTF_8));
   }
   ```
3. Author Python `feature_hasher.py` mirroring the canonicalization rules. Both implementations consume `feature_pipeline.json` and produce the same hash.
4. Modify `RegistryService.register()`:
   ```java
   String hash = hasher.compute(req.featurePipelinePath());
   var production = repo.findProductionPipeline(req.modelName());
   if (production.isEmpty()) {
     // First model ‚Äî bootstrap
     log.info("Bootstrap feature schema for {} = {}", req.modelName(), hash);
   } else if (!production.get().equals(hash)) {
     throw new FeatureSchemaMismatch(production.get(), hash, diffPaths(...));
   }
   // ... insert with feature_schema_hash = hash
   ```
   Where `findProductionPipeline` returns the first registered hash for that model_name (i.e., the canonical hash all subsequent versions must match).
5. Author `registerWithBootstrap(modelName, version, ...)`: takes a `ResetFeatureSchemaConfirmation` parameter. Archives all prior versions and registers as bootstrap.
6. Author parity tests:
   - `test_feature_hasher.py`: given a fixture `feature_pipeline.json`, Python returns expected hash.
   - `FeatureSchemaParityIT` (Java): same fixture, same hash.
7. Author `RegistryServiceIT` extension: registering V2 with mismatched pipeline ‚Üí throws.

## Known edge cases

- **Float formatting**: Python `json.dumps` may emit `1.0` while Java may emit `1`. Canonical form: always trailing `.0` for floats; use `Double.toString` after normalizing.
- **Map ordering**: both implementations sort keys alphabetically.
- **Nested arrays**: order preserved (semantic).
- **Unicode characters**: encode as `\uXXXX` for non-ASCII (lower-case hex).
- **Trailing newline**: hashed content has NO trailing newline; both implementations strip.
- **Bootstrap race**: two concurrent `register` calls for the same first-time model_name. Use a unique constraint or row-level lock; second caller fails with `DuplicateVersion`.
