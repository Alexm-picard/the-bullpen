# 3b.4 — `experiment_results` + promotion criteria

> Owning phase: `phase-3-ml-wrapper/3b-ab-routing` · Estimated effort: `5–7 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:

- Service for `experiment_results` (table from 3a.1)
- Pre-declared experiment: `start(modelName, championId, challengerId, primaryMetric, threshold, sampleSize, guardrails)`
- Compute experiment outcome from `prediction_log` joined with observed outcomes (for now, point estimates without statistical sequential analysis)
- `experiment.evaluate(experimentId)` reads paired predictions (champion + shadow on same request) and computes metric delta
- `experiment.complete(experimentId)` sets `status='passed'|'failed'` based on threshold + guardrails
- Used by 3a.4's promotion gate (already wired)
- Admin endpoints `POST /v1/admin/experiments/...`

**OUT of scope**:

- Sequential testing (mSPRT, Bayesian) → v1.5
- Frontend UI → Phase 4e.3

---

## Objectives

1. An experiment can be started, evaluated, and completed via API.
2. Promotion to champion is gated by a `passed` row (from 3a.4).
3. Eval reads paired predictions: same `request_id`, both champion and shadow rows present in `prediction_log`.
4. Guardrails enforced: any guardrail metric regressing causes `failed`.

## Step-by-step implementation tasks

1. Author DTOs:
   ```java
   public record StartExperimentRequest(
     String modelName, long championId, long challengerId,
     PrimaryMetric primaryMetric,        // BRIER | LOG_LOSS | ECE
     double primaryThreshold,            // e.g., 0.005 (challenger must beat by ≥)
     long sampleSizeTarget,
     Map<String, Double> guardrails      // metricName → max-allowed-delta
   ) {}
   ```
2. Author `ExperimentService.start(...)` — inserts a row with status=`running`.
3. Author `ExperimentService.evaluate(experimentId)`:
   - Counts paired predictions in `prediction_log` since `started_at` for the experiment's models.
   - Computes Brier (or specified primary) for each (champion, challenger) on observed outcomes (joined to `pitches.events` or analogous truth source).
   - Computes guardrail deltas (calibration error, p99 latency, per-class log loss).
   - Returns a `Verdict` (running, would-pass, would-fail) without mutating state.
4. Author `ExperimentService.complete(experimentId)`:
   - Calls `evaluate`.
   - If `sample_size_observed >= sample_size_target` AND verdict is `would-pass`: set `passed`.
   - Else if guardrail violated: set `failed`.
   - Else if `sample_size_observed < target`: refuse to complete (return error).
5. Author admin controller:
   - `POST /v1/admin/experiments/start`
   - `POST /v1/admin/experiments/{id}/evaluate`
   - `POST /v1/admin/experiments/{id}/complete`
6. Author tests:
   - Insert synthetic `prediction_log` paired rows where challenger is clearly better (Brier delta = 0.01 vs threshold 0.005); evaluate → would-pass; complete → passed.
   - Same but with guardrail violation (calibration regressed) → failed.
   - Insufficient sample size → complete refuses.

## Known edge cases

- **Truth source**: for the pitch outcome model, "observed truth" is the `pitches.description` (5-class). For batted-ball, it's `pitches.events` mapped to (out/1B/2B/3B/HR). Document the join semantics.
- **Late-arriving outcomes**: events for late games may not be in `pitches` until the nightly transform. Evaluate uses `request_at < now() - 24h` to ensure outcomes are present.
- **Aborted state**: a `kill` admin endpoint can abort a running experiment without a verdict (e.g., regretted starting it).
- **Multiple concurrent experiments per model_name**: forbid in v1; one running experiment per model_name.
