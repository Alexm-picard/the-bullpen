# 3b.1 — Routing config schema + service

> Owning phase: `phase-3-ml-wrapper/3b-ab-routing` · Estimated effort: `3–4 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Service layer over `model_routing` (table from 3a.1)
- `RoutingService.getRouting(modelName) -> RoutingConfig`
- `RoutingService.setChallenger(modelName, challengerVersionId, mode)` — used by 3a.4 as part of `transitionStage(SHADOW)`
- `RoutingService.setTrafficPct(modelName, pct)` — admin-only
- Cache layer: routing config read-heavy on the inference path; cache with 30s TTL or Spring Caching abstraction
- Validation: a candidate cannot be both champion and challenger; setting challenger requires it to be in `shadow` stage

**OUT of scope**:
- Murmur3 bucketing → `3b.2`
- Shadow-mode dispatch → `3b.3`
- Frontend admin UI → Phase 4e.3

---

## Objectives

1. `getRouting` returns a fresh-enough `RoutingConfig` in <1ms (cached).
2. Setting traffic_pct outside [0,100] rejected.
3. Setting challenger to a `champion`-stage version rejected.
4. Cache invalidates on writes; admin slider changes visible within 30s.

## Step-by-step implementation tasks

1. Author `RoutingConfig` record: `(championVersionId, challengerVersionId?, challengerTrafficPct, mode)`.
2. Author `RoutingRepository`: `findByModelName`, `upsert`. JdbcTemplate-backed.
3. Author `RoutingService`:
   ```java
   @Cacheable(value="routing", key="#modelName")
   public RoutingConfig getRouting(String modelName) { return repo.findByModelName(modelName).orElseThrow(); }

   @CacheEvict(value="routing", key="#modelName")
   public void setChallenger(String modelName, long challengerId, RoutingMode mode) {
     var ch = registry.getById(challengerId);
     if (ch.stage() != Stage.SHADOW) throw new IllegalArgumentException("...");
     repo.upsert(modelName, ..., 0.0, mode);
   }

   @CacheEvict(value="routing", key="#modelName")
   public void setTrafficPct(String modelName, double pct) {
     if (pct < 0 || pct > 100) throw new IllegalArgumentException();
     // ...
   }
   ```
   Configure Caffeine cache with 30s TTL.
4. Author `RoutingAdminController`:
   - `POST /v1/admin/routing/{name}/challenger`
   - `POST /v1/admin/routing/{name}/traffic-pct`
5. Author `RoutingServiceIT`:
   - Set challenger to a SHADOW model → success.
   - Set challenger to a CHAMPION model → fail.
   - Out-of-range pct → fail.
   - Cache hit/miss counters increment as expected.

## Known edge cases

- **Cache stampede**: Caffeine with `refresh-after-write` prevents on-cold-cache thundering herd. Configure once.
- **Mode = `'shadow'` with traffic_pct > 0**: should be rejected (in shadow mode, no traffic to challenger). Validate in service.
- **First-ever model with no routing row**: `getRouting` returns the only registered champion's config implicitly. Or eagerly insert a routing row on first promotion to champion.
