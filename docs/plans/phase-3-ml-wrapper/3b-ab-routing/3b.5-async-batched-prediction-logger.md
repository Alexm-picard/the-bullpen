- Replace 1.7's primitive logger with the full version

## Step-by-step implementation tasks

1. V015:
   ```sql
   ALTER TABLE prediction_log
     MODIFY COLUMN role Enum8('champion'=1,'challenger'=2,'shadow'=3),
     ADD COLUMN IF NOT EXISTS feature_hash String AFTER model_version_id,
     COMMENT COLUMN features 'JSON-encoded; columnar variant in 3b.6 candidate v1.5';
   -- TTL set in 1.7 already; no change.
   ```
2. Author `PredictionLogEventV2` record with all V2 fields. Mark V1 record `@Deprecated`.
3. Update `AsyncPredictionLogger`:
   - Accept V2 events.
   - Increase queue size to 20K (was 10K) to absorb shadow-mode 2× volume.
4. Update `PredictionLogWriter` to insert V2 columns.
5. Update `InferenceRouter` (from 3b.3) to construct V2 events with correct `role` and `model_version_id`.
6. Backfill / migration consideration: old V1 rows (where role wasn't set) get `'champion'` via DEFAULT — verify in V015.
7. Tests:
   - `AsyncPredictionLoggerV2Test`: enqueue 100 V2 events; all flushed.
   - `PredictionLogWriterV2IT`: V2 INSERT against Testcontainers.
   - Backwards-compat: query a V1 row → returns with `role='champion'`.

## Known edge cases

- **Schema rollback**: if you need to roll back V015, you'd lose the new columns' data. Forward-only rule applies — don't roll back.
- **`features` as JSON String column**: still stringy in v1; columnar restructure deferred to v1.5.
- **Backfill of `feature_hash` for V1 rows**: leave NULL. Drift computations skip NULLs.
- **Volume**: shadow mode + post-pitch model means 4× request volume → 4× rows. Verify TTL still leaves manageable disk.
