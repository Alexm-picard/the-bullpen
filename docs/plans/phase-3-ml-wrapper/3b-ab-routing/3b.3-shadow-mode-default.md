- Wires into both `PredictPitchController` and `PredictBattedBallController`

**OUT of scope**:

## Objectives

1. Shadow mode runs both models on every request; user sees only champion.
2. Both predictions logged to `prediction_log` with correct `role`.
3. Shadow latency does NOT inflate user-facing latency (parallel execution).
4. Warm-up gates readiness probe until both models are loaded.

## Step-by-step implementation tasks

1. Author `ModelLoader`:
   - Loads ONNX, calibrator, pipeline lookups for any registered model version.
   - Caches loaded models in memory (Caffeine, max 4 versions per model_name).
2. Author `InferenceRouter`:

   ```java
   public PredictionResponse predict(Object req, String modelName, long gameId) {
     RoutingConfig cfg = routing.getRouting(modelName);
     Role primaryRole = bucketer.route(gameId, modelName, cfg);

     // Always run champion (it's what users see)
     CompletableFuture<float[]> championFut = supplyAsync(() -> models.get(cfg.championVersionId()).predict(req));

     // In shadow mode: also run challenger
     // In A/B mode: if primaryRole==CHALLENGER, only run challenger
     CompletableFuture<float[]> shadowFut = (cfg.mode() == SHADOW && cfg.challengerVersionId() != null)
       ? supplyAsync(() -> models.get(cfg.challengerVersionId()).predict(req))
       : CompletableFuture.completedFuture(null);

     float[] returned = (cfg.mode() == AB && primaryRole == Role.CHALLENGER)
       ? shadowFut.join() : championFut.join();

     // Log both
     enqueueLog(cfg.championVersionId(), Role.CHAMPION, championFut.getNow(null));
     if (cfg.mode() == SHADOW && cfg.challengerVersionId() != null) {
       enqueueLog(cfg.challengerVersionId(), Role.SHADOW, shadowFut.getNow(null));
     } else if (cfg.mode() == AB && cfg.challengerVersionId() != null && primaryRole == Role.CHALLENGER) {
       enqueueLog(cfg.challengerVersionId(), Role.CHALLENGER, shadowFut.getNow(null));
     }

     return PredictionResponse.from(returned, ...);
   }
   ```

3. Wire into both controllers.
4. Extend `WarmupReadiness` to warm up all `(champion, challenger)` pairs across all registered model_names.
5. Author `InferenceRouterIT`:
   - Shadow mode: assert two log rows per request (CHAMPION + SHADOW).
   - A/B mode (50%): over 1000 requests, assert ~500 go to CHAMPION, ~500 to CHALLENGER.
   - Latency: shadow mode p95 within 10% of single-model p95.

## Known edge cases

- **Challenger model fails to load**: shadow mode silently degrades to champion-only (logs a NOTICE). Don't fail the user request.
- **Challenger prediction throws**: catch + log NOTICE; keep championFut result.
- **Concurrent `setChallenger` during a flight of requests**: cache invalidation handles. Brief inconsistency acceptable (â‰¤30s).
- **Different feature pipelines**: champion and challenger MUST share the same `feature_schema_hash` (enforced by 3a.3). If they don't, registration would have rejected. Sanity-check at load time anyway.
- **Async logger overflow**: 2Ă— the row volume in shadow mode. Verify the queue size from 1.7 still suffices; bump if needed.
