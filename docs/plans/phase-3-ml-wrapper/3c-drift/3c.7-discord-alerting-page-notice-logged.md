# 3c.7 — Discord alerting (page / notice / logged-only)

> Owning phase: `phase-3-ml-wrapper/3c-drift` · Estimated effort: `4–5 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:

- `DriftAlertEvaluator` Spring component reading `drift_metrics` and applying the threshold table from `00-OBSERVABILITY-STRATEGY.md`
- Page-worthy: champion calibration error > 1.5× training calibration sustained 3+ days → page
- Notice-worthy: feature PSI > 0.25 sustained 7+ days → notice
- Everything else: logged-only (visible on Ops dashboard, no Discord ping)
- Deduplication: if the same alert fired in the past 24h, suppress (avoid Discord spam)
- Runbook links embedded in alert messages

**OUT of scope**:

- The drift jobs themselves → 3c.2–3c.5
- Frontend display of logged-only events → Phase 4e.2

---

## Objectives

1. Page-worthy events fire to Discord with the agreed template.
2. Notice-worthy events fire to Discord (different visual prefix).
3. Logged-only events do NOT fire to Discord but are queryable for the dashboard.
4. Deduplication: same alert key within 24h suppressed.

## Step-by-step implementation tasks

1. V014:
   ```sql
   CREATE TABLE alert_history (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     alert_key TEXT NOT NULL,                 -- e.g., 'pitch_outcome_pre/calibration_error/all'
     severity TEXT NOT NULL CHECK (severity IN ('PAGE','NOTICE','LOGGED')),
     fired_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
     metric_value REAL,
     metric_threshold REAL,
     details TEXT
   );
   CREATE INDEX idx_ah_key_time ON alert_history(alert_key, fired_at);
   ```
2. Author `DriftAlertEvaluator`:

   ```java
   @Scheduled(cron = "0 0 3 * * *", zone = "America/New_York")
   public void evaluate() {
     for (var model : registry.activeModels()) {
       // PAGE: calibration > 1.5x training cal sustained 3 days
       var calRecent = repo.findLast3Days(model.name(), MetricType.CALIBRATION_ERROR, "all");
       double trainingCal = readTrainingCalibration(model);
       if (calRecent.size() == 3 && calRecent.stream().allMatch(m -> m.value() > 1.5 * trainingCal)) {
         fireIfNotRecent(PAGE, "drift/" + model.name() + "/calibration", "Calibration drift on " + model.name(), ...);
       }
       // NOTICE: feature PSI > 0.25 sustained 7 days
       for (var feature : model.features()) {
         var psiRecent = repo.findLast7Days(model.name(), MetricType.PSI_FEATURE, feature);
         if (psiRecent.size() >= 7 && psiRecent.stream().allMatch(m -> m.value() > 0.25)) {
           fireIfNotRecent(NOTICE, "drift/" + model.name() + "/psi/" + feature, ...);
         }
       }
     }
   }

   private void fireIfNotRecent(Severity sev, String key, ...) {
     if (alertHistory.firedWithin(key, Duration.ofHours(24))) return;
     discord.send(sev, ..., Map.of("runbook", runbookUrl(sev, key), "trace_id", MDC.get("traceId")));
     alertHistory.record(key, sev, ...);
   }
   ```

3. Author `AlertHistory` repository (SQLite).
4. Document runbooks: `calibration-drift-investigation.md`, `feature-drift-investigation.md`.
5. Author `DriftAlertEvaluatorIT`:
   - Insert 3 days of calibration metrics > 1.5x training → PAGE fires.
   - Same conditions on day 4 → suppressed (dedup).
   - Insert 7 days of PSI > 0.25 → NOTICE fires.
   - Mixed-day metrics not all > threshold → no fire.

## Known edge cases

- **`training_calibration` not in metadata.json**: fall back to using the first 7 days of post-deploy calibration as baseline. Document.
- **Discord rate limits**: if many alerts fire in burst (e.g., post-rollout regression), Discord may rate-limit. Implement a 1/sec flusher in DiscordAlerter (a Phase 0.9 follow-up if needed).
- **Time zones in dedup**: use UTC for `fired_at` comparison; presentation in dashboards is ET.
- **Holiday off-season silence**: from November to February, no games → calibration not evaluable. Job exits early with a NOTICE saying "no recent predictions to evaluate". Don't fire false positives.
