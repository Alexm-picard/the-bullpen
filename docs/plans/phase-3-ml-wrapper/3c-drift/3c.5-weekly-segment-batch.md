# 3c.5 — Weekly segment metrics

> Owning phase: `phase-3-ml-wrapper/3c-drift` · Estimated effort: `5–7 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Sunday-night Spring `@Scheduled` job computing per-segment calibration metrics
- Segments: handedness, park_id, count state, inning bucket, pitch_type, day-of-week, month-of-season
- Long-window comparisons: 7d / 28d / season-to-date
- Outcome rate vs. predicted rate per segment
- Writes `drift_metrics` rows with `metric_type=SEGMENT_BRIER`

**OUT of scope**:
- Real-time alerting → 3c.7
- Visualization → Phase 4e.2

---

## Objectives

1. Weekly job runs Sunday 23:30 ET, completes in <30 minutes.
2. ~50 segment metrics × 4 windows = ~200 metric rows per active model per run.
3. Detects within-season non-stationarity (decision `[61]`).

## Known edge cases

- **Pitch_type "UN"**: unclassified pitches; treat as own segment.
- **Off-season (Nov–Mar)**: jobs run but find no recent data. Write rows with `sample_size=0`. Dashboard skips zero-sample.
- **Postseason**: `month` values 10–11 may have skewed metrics due to pitcher quality. Acceptable; document.
- **Memory pressure on large windows**: 220-day window may be ~2M predictions for an active model. Stream the SQL aggregation; don't pull all rows into Java memory.
