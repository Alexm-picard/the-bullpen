# 4e.1 — Ops: registry browser

> Owning phase: `phase-4-frontend/4e-ops-dashboard` · Estimated effort: `4–6 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Read-only public page at `/ops` with a table of all model versions, filterable by `model_name` and `stage`
- Each row shows: model_name, version, stage, trained_at, promoted_at, primary metrics summary
- Click a row → drawer/modal with full eval metrics (parsed from `eval_metrics` JSON column)
- Recruiter-facing page (decision `[99]`) — never cut

**OUT of scope**:
- Drift charts → 4e.2
- A/B status → 4e.3
- Retrain queue → 4e.4
- Reliability diagrams → 4e.5

---

## Objectives

1. `/ops` loads in <500 ms.
2. Table is sortable, filterable, and paginated (50 rows per page).
3. Detail drawer shows full eval metrics in a readable layout.

## Step-by-step implementation tasks

1. Author hook `useRegistry(modelName?, stage?)` calling `/v1/ops/registry/...` (or a new `/v1/ops/registry` aggregation endpoint that returns all model_names).
2. Author `<ModelVersionTable />` using Mantine's `Table`. Sort by `trained_at` desc by default. Filter dropdowns for model_name and stage.
3. Author `<ModelVersionDetail />` drawer:
   - Metadata: model_name, version, stage, training data window, trained_at, promoted_at, training_data_hash, feature_schema_hash.
   - Eval metrics: pretty-printed summary (mean ± std per fold).
4. Add an `<Tabs />` skeleton on `/ops` with five tabs corresponding to 4e.1–4e.5; this leaf only fills the Registry tab.
5. Tests: Playwright loads the page, expects to see at least one model version row.

## Known edge cases

- **Empty registry**: pre-Phase-3 deployment shows "No models registered yet". Don't crash.
- **`eval_metrics` JSON shape varies** by model type. Detail drawer renders generically (key-value table); special handling for known keys (Brier, ECE) with units.
- **Long model_name list**: support 30+ model names via search-as-you-type filter.
