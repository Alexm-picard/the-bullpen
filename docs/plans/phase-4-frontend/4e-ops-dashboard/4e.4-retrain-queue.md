## Scope boundaries

**IN scope**:

- Retrain tab on `/ops` showing the contents of `retraining_queue`
- Status counts (queued / running / succeeded / failed) + recent 20 entries
- Per-row: trigger_type, trigger_id, model_name, enqueued_at, started_at, finished_at, error_message
- Manual trigger button (admin-only) opening a modal: select model_name ‚Üí enqueue ‚Üí row appears

**OUT of scope**:

- Cancellation UI for running jobs ‚Äî admin-only API call exists; surface in v1.5
- Detailed log viewer per run ‚Äî links to journalctl/grafana instead

2. Manual-trigger modal works for admin users.
3. Auto-refresh every 30 s; new rows appear.

## Step-by-step implementation tasks

1. Backend: `GET /v1/ops/retraining-queue?limit=20` ‚Äî public endpoint returning recent rows.
2. Frontend: `useRetrainQueue()` polls every 30 s.
3. `<RetrainQueueTable />` ‚Äî Mantine table with status badges (queued: gray, running: blue, succeeded: viridis-green, failed: brick-red accent).
4. `<AdminManualRetrainModal />`:
   - Dropdown of active model_names from registry.
   - Submit calls `POST /v1/admin/retrain` with admin auth.
   - On success, closes modal and forces a queue refetch.
5. Tests: Vitest snapshot of table; Playwright loads tab + opens modal (disabled without admin auth).

## Known edge cases

- **`failed` rows pile up**: show recent 20; older filtered. If many fails, surface a NOTICE banner.
- **Stale `running` rows**: stale-claim reaper from 3d.4 catches; UI just reflects DB state.
- **Empty queue**: empty state ("No retrain runs yet").
