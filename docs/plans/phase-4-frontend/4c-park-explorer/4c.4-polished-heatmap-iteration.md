## Step-by-step implementation tasks

1. Build the Viridis colormap helper (lookup table 256 stops).
2. Replace the single-tint `<rect>` overlay with a Viridis-mapped fill based on P(HR).
3. Compute landing zone per park: simple deterministic computation from the canonical input + park-mean conditions (or just take the median of physics simulator output).
4. Render a small dot at the landing point on each SVG.
5. Reorder grid client-side by descending P(HR) on every refetch.
6. Author `<ParkDetailPanel />`: hovering a thumbnail expands a small panel showing 5-class probability bars (P(out), P(1B), P(2B), P(3B), P(HR)), plus the park name in display serif.
7. Optimize re-render: memoize thumbnails so only the changed ones re-render. Use `React.memo` keyed on `(parkId, probHr)`.
8. Profile: `<ProfilerProvider />` measures render duration; assert <16ms per update on a mid-tier laptop.
9. Visual review: take screenshots; iterate until the page reads "editorial-data" not "SaaS chrome".

## Known edge cases

- **Reordering jitter**: when HR probabilities are close, tiny slider moves can reorder the grid. Add a stable sort with a secondary key (alphabetical park id) to dampen.
- **Mobile viewport**: 6×5 grid → 5 cols on desktop, 3 cols on tablet, 2 cols on phone. Mantine breakpoints handle.
- **Color contrast on light viridis stops**: lightest stops can clash with the warm off-white background. Outline thumbnails with a 1px border (decision `[107]`: borders not shadows).
- **Hover on touch**: simulate via `onTouchStart`. Detail panel auto-dismisses after 3s.
