# 4d.1 — TanStack polling (10–15 s) for live games

> Owning phase: `phase-4-frontend/4d-game-live` · Estimated effort: `5–7 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- Backend: live polling worker job populates `pitches_live` table from MLB Stats API (game state machine — Risk Register G5)
- Backend endpoint `GET /v1/games/today` returning today's games + their state
- Backend endpoint `GET /v1/games/:id/pitches?since=` returning pitches added since a cursor
- Frontend: `/game/:id` page polling at 10–15 s interval via TanStack Query
- Cache invalidation pattern using `since=<lastPitchId>` so polls only fetch deltas

**OUT of scope**:
- Pitch-by-pitch UI with predictions overlay → 4d.2
- Game state machine logic detail — same as Risk Register G5 resolution

**Soft cut**: Phase 4d (this entire sub-tree) is the FOURTH soft-cut candidate.

---

## Objectives

1. Live game endpoint serves under 200 ms p95 even at 10s polling.
2. Game state machine handles `in_progress`, `mid_inning`, `rain_delay`, `suspended`, `postponed`, `completed`.
3. Polling resumes correctly after browser tab regains focus (TanStack Query default).
4. Live polling lag < 30 s from MLB API to ClickHouse (Risk Register I1 SLO).

## Step-by-step implementation tasks

1. V018 — `pitches_live`:
   ```sql
   CREATE TABLE pitches_live (
     game_id UInt64, at_bat_index UInt16, pitch_number UInt8,
     game_date Date, ingested_at DateTime DEFAULT now(),
     -- minimal columns; live data sparse
     pitcher_id UInt32, batter_id UInt32,
     description LowCardinality(String),
     pitch_type LowCardinality(String) DEFAULT '',
     release_speed_mph Nullable(Float32),
     plate_x_in Nullable(Float32), plate_z_in Nullable(Float32),
     balls UInt8, strikes UInt8, outs UInt8, inning UInt8,
     home_score UInt8, away_score UInt8
   ) ENGINE = ReplacingMergeTree(ingested_at)
   PARTITION BY toYYYYMM(game_date)
   ORDER BY (game_id, at_bat_index, pitch_number);
   ```
2. Author `MlbStatsApiClient` — REST client with rate-limit token bucket (2 req/s). Endpoints:
   - `GET https://statsapi.mlb.com/api/v1/schedule?date=YYYY-MM-DD&sportId=1`
   - `GET https://statsapi.mlb.com/api/v1.1/game/{gamePk}/feed/live`
3. Author `GameStateMachine` (Risk Register G5 resolution): state transitions per `gameData.status.detailedState`; per-state polling intervals (in_progress: 12s, warmup: 60s, others: off).
4. Author `LivePollingService` (`@Profile("worker")`):
   - `@Scheduled(fixedDelay=60000)` — checks today's schedule, starts/stops per-game pollers.
   - For each in_progress game: fire a virtual-thread poll at the game's interval.
   - Insert new pitches into `pitches_live`.
   - Emit metric `live_poll_lag_seconds{game_id=...}`.
5. Author `GameController`:
   - `GET /v1/games/today` → list of today's games.
   - `GET /v1/games/{id}` → game summary.
   - `GET /v1/games/{id}/pitches?since=<lastPitchId>` → delta of new pitches.
6. Frontend: hooks
   ```ts
   useGame(id) // refetchInterval based on gameStatus
   useLivePitches(id) // queries with since=<lastPitchId>; merges into local list
   ```
7. Author `<GamePage />` showing game header + a placeholder pitch list (4d.2 polishes).
8. Author state-machine tests covering all transitions + edge cases (doubleheader, rain delay → resumed, postponed mid-game).

## Known edge cases

- **Network blip / 5xx from MLB**: token bucket + exponential backoff; never crash.
- **Doubleheader**: schedule lookup re-runs at 1 PM and 5 PM ET to catch the second game (Risk Register G5).
- **Postponed game**: state transition `in_progress → postponed`; polling stops; the next-day schedule fetches the rescheduled game.
- **Rate limit headers**: MLB API may return 429. Detect and back off heavier (60s sleep).
- **`pitches_live` retention**: keep 14 days; overnight handoff job moves to canonical `pitches` (Phase 3 ingest leaf — out of scope here, future leaf).
