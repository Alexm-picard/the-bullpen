# 1.6 — Park Explorer toy page

> Owning phase: `phase-1-vertical-slice` · Estimated effort: `4–6 h` · Authored: 2026-05-10

---

## Scope boundaries

**IN scope**:
- React page at `/parks` rendering ~10 hardcoded historical batted-ball examples (curated as a static JSON asset)
- Each entry shows the inputs (launch speed, launch angle, park) and a "Predict" button
- Clicking "Predict" calls `POST /v1/predict/batted-ball` and renders the returned `probHr`
- Loading skeleton + error state + empty state visible
- Uses TanStack Query mutations for the predict call
- A single Playwright e2e test (`vertical-slice.spec.ts`) covers the click → response cycle

**OUT of scope**:
- 30-stadium heatmap → Phase 4c (the marquee component)
- Sliders to vary input parameters → Phase 4c.3
- Real design tokens / typography → Phase 4a (this leaf uses plain Tailwind)

---

## Objectives

1. Visit `https://thebullpen.net/parks` from any device → page loads.
2. Click any of the 10 example balls → real prediction renders.
3. Network failure / 500 → error state visible, no console crash.
4. Vertical-slice e2e test passes locally + in CI.

## Step-by-step implementation tasks

1. Add deps:
   ```bash
   cd frontend
   pnpm add react-router-dom
   pnpm add -D @playwright/test
   pnpm exec playwright install chromium
   ```
2. Wire router in `App.tsx` with two routes: `/` (the existing health page) and `/parks`.
3. Author `data/example-bbips.json` — 10 entries, each with:
   ```json
   { "id": "ex-1", "label": "Aaron Judge HR @ Yankee Stadium",
     "launchSpeedMph": 117.4, "launchAngleDeg": 31, "releaseSpeedMph": 95, "parkId": "NYA", "stand": "R" }
   ```
   Curate from real 2024 events (varied parks + outcomes — at least 4 HRs, 4 outs, 2 ambiguous).
4. Author `use-predict-batted-ball.ts`:
   ```ts
   import { useMutation } from '@tanstack/react-query';
   import type { components } from './types';
   type Req = components['schemas']['BattedBallRequest'];
   type Res = components['schemas']['PredictionResponse'];
   export const usePredictBattedBall = () =>
     useMutation<Res, Error, Req>({ mutationFn: (req) => post('/predict/batted-ball', req) });
   ```
5. Author `example-bbip-card.tsx` — renders a single example. Has a "Predict" button. On click, calls `mutate()`. Renders `data?.probHr`, `error?.message`, or skeleton based on `status`.
6. Author `parks-page.tsx` — maps over the 10 examples, renders cards in a grid. Uses Tailwind for layout (no custom design tokens yet).
7. Author `playwright.config.ts`:
   ```ts
   import { defineConfig } from '@playwright/test';
   export default defineConfig({
     testDir: './tests',
     use: { baseURL: process.env.E2E_BASE_URL ?? 'http://localhost:5173' },
     webServer: process.env.E2E_BASE_URL ? undefined : {
       command: 'pnpm dev',
       url: 'http://localhost:5173',
       reuseExistingServer: !process.env.CI,
     },
     projects: [{ name: 'chromium', use: { ...devices['Desktop Chrome'] } }],
   });
   ```
8. Author `vertical-slice.spec.ts`:
   ```ts
   import { test, expect } from '@playwright/test';
   test('park explorer renders predictions', async ({ page }) => {
     await page.goto('/parks');
     await expect(page.getByRole('heading', { name: /parks/i })).toBeVisible();
     await page.getByTestId('example-bbip-ex-1').getByRole('button', { name: /predict/i }).click();
     await expect(page.getByTestId('example-bbip-ex-1')).toContainText(/p\(HR\)/i);
   });
   ```
9. Author `.github/workflows/e2e-smoke.yml`:
   ```yaml
   name: e2e-smoke
   on: [push, pull_request]
   jobs:
     e2e:
       runs-on: ubuntu-latest
       defaults: { run: { working-directory: frontend } }
       steps:
         - uses: actions/checkout@v4
         - uses: pnpm/action-setup@v3
           with: { version: 9 }
         - uses: actions/setup-node@v4
           with: { node-version: 20, cache: pnpm, cache-dependency-path: frontend/pnpm-lock.yaml }
         - run: pnpm install --frozen-lockfile
         - run: pnpm exec playwright install --with-deps chromium
         - name: Start backend (mock)
           working-directory: frontend
           run: echo "Frontend uses MSW or mock backend for e2e in CI"
         - run: pnpm exec playwright test
   ```
   For Phase 1 simplicity, use MSW (Mock Service Worker) to intercept `/v1/predict/batted-ball` in the dev/CI environment — avoids spinning the full Java backend in the e2e job.
10. Push to `main`. Verify Vercel deploys; visit `/parks`; click a card; see `probHr`.

## Known edge cases

- **CORS preflight on POST**: the existing `CorsConfig` from 0.6 must include `POST` (it does) and accept `Content-Type: application/json` — confirm.
- **Stand `'S'`**: hand-curated examples should avoid switch hitters until Phase 2.
- **MSW vs. real backend in CI**: MSW is fine for the vertical-slice test. Phase 4 will introduce real-backend e2e when the Java API runs in CI as a service container.
- **Page bundle size**: this page alone shouldn't exceed ~20 KB gz. If it does, something's wrong.
- **React Router `BrowserRouter` on Vercel**: needs a `vercel.json` rewrite for SPA routing — add it now (`{"rewrites":[{"source":"/(.*)","destination":"/"}]}`).
