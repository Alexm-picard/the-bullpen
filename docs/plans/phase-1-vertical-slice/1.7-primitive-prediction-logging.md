## Scope boundaries

**IN scope**:
- ClickHouse `prediction_log` table (Phase 1 baseline schema)
- Async batched logger: bounded in-memory queue + background flusher writing every N seconds or M rows (whichever first)
- Drop-on-overflow policy with Micrometer counter (`prediction_log_dropped_total`)
- Queue-depth gauge (`prediction_log_queue_depth`)
- Wired from `PredictBattedBallController`: every prediction enqueues a log event
- TTL on partitions = 18 months (Risk Register G8)

**OUT of scope**:
- `role` column with shadow/challenger semantics ‚Üí Phase 3b.5 will alter the table
- `experiment_results` join ‚Üí Phase 3b
- Reconciliation job for cross-DB integrity ‚Üí Phase 3a.5

## Objectives

1. Every successful `/v1/predict/batted-ball` produces a row in `prediction_log` within 5 seconds.
2. Killing the API mid-burst loses ‚Č§10 predictions (queue size bound).
3. Drop counter increments under sustained overload; service stays healthy.
4. ClickHouse partitioning + TTL configured.

## Step-by-step implementation tasks

1. Author `V004__prediction_log.sql`:
   ```sql
   CREATE TABLE IF NOT EXISTS prediction_log (
     request_id        UUID,
     request_at        DateTime64(3, 'UTC'),
     model_name        LowCardinality(String),
     model_version_id  UInt32,
     role              Enum8('champion'=1,'challenger'=2,'shadow'=3) DEFAULT 'champion',
     feature_hash      String,
     features          String,           -- JSON-encoded; full versioning in Phase 3b
     prediction        String,           -- JSON-encoded
     latency_ms        Float32,
     trace_id          String
   ) ENGINE = MergeTree()
   PARTITION BY toYYYYMM(request_at)
   ORDER BY (model_name, request_at)
   TTL toDate(request_at) + INTERVAL 18 MONTH;
   ```
2. Author `PredictionLogEvent` record.
3. Author `PredictionLogWriter`:
   ```java
   @Component
   public class PredictionLogWriter {
     private final NamedParameterJdbcTemplate clickhouse;
     public void writeBatch(List<PredictionLogEvent> batch) {
       String sql = "INSERT INTO prediction_log (...) VALUES (...)";
       jdbcTemplate.batchUpdate(sql, batch.stream().map(this::toParams).toList());
     }
   }
   ```
4. Author `AsyncPredictionLogger`:
   ```java
   @Component
   @Profile("api")
   public class AsyncPredictionLogger {
     private final BlockingQueue<PredictionLogEvent> queue = new ArrayBlockingQueue<>(10_000);
     private final ScheduledExecutorService flusher = Executors.newSingleThreadScheduledExecutor();
     private final Counter dropped;
     private final Gauge depth;
     @PostConstruct void start() {
       flusher.scheduleAtFixedRate(this::flush, 1, 1, TimeUnit.SECONDS);
     }
     public void enqueue(PredictionLogEvent ev) {
       if (!queue.offer(ev)) dropped.increment();
     }
     private void flush() {
       List<PredictionLogEvent> batch = new ArrayList<>(500);
       queue.drainTo(batch, 500);
       if (batch.isEmpty()) return;
       try { writer.writeBatch(batch); } catch (Exception e) { log.warn("write failed; events dropped", e); dropped.increment(batch.size()); }
     }
     @PreDestroy void stop() {
       flusher.shutdown(); flusher.awaitTermination(10, TimeUnit.SECONDS); flush();
     }
   }
   ```
   Use a virtual-thread executor where possible (`Executors.newSingleThreadScheduledExecutor(Thread.ofVirtual().factory())` is fine but a regular scheduler is simpler here ‚ÄĒ the flusher is one thread).
5. Wire into `PredictBattedBallController.predict()` after computing the prediction:
   ```java
   logger.enqueue(new PredictionLogEvent(
     UUID.randomUUID(), Instant.now(), "_toy_batted_ball", 0,
     PredictionLogEvent.Role.CHAMPION, featureHash, featureJson, predJson,
     latencyMs, MDC.get("traceId")));
   ```
6. Author `AsyncPredictionLoggerTest`:
   - Enqueue 100 events; assert all flushed within 2s.
   - Saturate beyond capacity (10001 events); assert drop counter >= 1, no exceptions.
7. Author `PredictionLogWriterIT` (Testcontainers ClickHouse):
   - Insert 50 events; `SELECT count(*) FROM prediction_log` returns 50.
8. Run a load test locally:
   ```bash
   hey -n 1000 -c 50 -m POST -T 'application/json' \
     -d '{"launchSpeedMph":105,"launchAngleDeg":28,"releaseSpeedMph":94,"parkId":"NYA","stand":"R"}' \
     http://localhost:8080/v1/predict/batted-ball
   ```
   Then confirm: `SELECT count(*) FROM prediction_log WHERE request_at > now() - INTERVAL 1 MINUTE`.
9. Negative test: stop ClickHouse for 30 seconds during traffic. Verify:
   - API still returns 200.
   - Drop counter increments.
   - When ClickHouse comes back, queue drains naturally.

## Known edge cases

- **`PreDestroy` flush race**: must call `flush()` after `awaitTermination` returns, OR the in-flight events are lost. Document.
- **Schema evolution**: Phase 3b.5 will add `role` semantics (we set DEFAULT to `'champion'` here ‚ÄĒ additive, safe). Will not break existing rows.
- **TTL precision**: `INTERVAL 18 MONTH` evaluated against `toDate(request_at)` so the entire month's partition rolls off at once ‚ÄĒ predictable disk usage.
- **`features` as JSON String column**: not great for compression; LowCardinality + structured columns will be retrofitted in 3b.5. For Phase 1, JSON is fine.
- **Backpressure-by-design**: dropping is the contract. Never make the predict endpoint wait on the logger.
