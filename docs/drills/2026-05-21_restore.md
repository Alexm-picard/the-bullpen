Tonight's pre-flight (step 1 of the planned drill) surfaced enough plumbing complexity to warrant pausing and doing the full drill with a proper artifact (`ops/scripts/restore-drill.sh`) tomorrow rather than improvising at 21:40 ET. Specifically:

1. The `clickhouse-backup` binary lives on the host but the ClickHouse data dir is in a Docker named volume (`thebullpen_clickhouse-data`), and the backups themselves land at `/var/lib/clickhouse/backup/` on the host while the FREEZE shadow data parts live inside the Docker volume at `/var/lib/docker/volumes/thebullpen_clickhouse-data/_data/shadow/`. A correct restore needs to bridge that split.
2. The `ops/scripts/restore-drill.sh` script referenced in `docs/plans/phase-0-foundation/0.10-backup-restore-drill.md` (the canonical spec for this drill) does NOT exist yet. Tomorrow's session will write it.
3. Both existing backups (`auto_20260520T011526Z`, `auto_20260521T234637Z`) are metadata-only (390 bytes each, `data:0B`) because the test table `bullpen._drill_marker` has its rows in memtables, not flushed to disk parts. The drill should either accept that limitation explicitly OR insert data + force a FLUSH before re-taking a snapshot.

Discipline-rule-8 framing: this drill is already overdue (MLB season started in April; today is mid-May). Doing it +24h later, with a proper artifact, doesn't change the "overdue" label.
