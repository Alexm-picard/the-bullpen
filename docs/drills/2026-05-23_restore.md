## Critical finding (Phase 0 has been silently broken since 2026-05-19)

The pre-existing snapshot path running `clickhouse-backup create` **on the host** was producing
metadata-only backups (390 bytes each, `data:0B`). Repro from `infra/backup/clickhouse-snapshot.sh`
v1: 3 of 3 auto-snapshots since 2026-05-19 captured zero data. The discipline-rule-8 framing was
right to flag the partial-drill as overdue evidence: a backup that has never been restore-tested is
not a backup, and in our case it literally wasn't even capturing the data.

**Root cause:** `clickhouse-backup` issues `ALTER TABLE … FREEZE WITH NAME '<hash>'` through the
ClickHouse client connection. ClickHouse writes the FREEZE shadow data to its own
`/var/lib/clickhouse/shadow/<hash>/` — which inside the container maps to the host's
`/var/lib/docker/volumes/thebullpen_clickhouse-data/_data/shadow/<hash>/`. But `clickhouse-backup`
running on the host then looks for the FREEZE output at the host's literal
`/var/lib/clickhouse/shadow/<hash>/` (a different directory entirely — the host has its own
stub `/var/lib/clickhouse/` directory next to the docker-volume mount), finds nothing, copies
nothing into the backup, and immediately UNFREEZEs. Result: backup directory has metadata
but no `shadow/` subdirectory with data parts.

The `data_path` config knob in `/etc/clickhouse-backup/config.yml` was set to the docker-volume
host path, but the v2.6.43 binary doesn't honor it for the shadow-source lookup — it trusts
ClickHouse's `system.disks.path` query instead, which returns the container-internal path.

**Fix:** run `clickhouse-backup` **inside** the ClickHouse container via `docker exec`. Then the
binary sees the same `/var/lib/clickhouse/shadow/` as ClickHouse itself, finds the FREEZE output,
and successfully copies the data parts into the backup directory. Verified: 770B data captured
vs 0B before.

Patched in this session:

- `infra/backup/clickhouse-snapshot.sh` — now runs `clickhouse-backup create` via
  `docker exec bullpen-clickhouse`, auto-installs the binary + config inside the container
  on first run, and adds a sanity check that fails loudly if a backup captures 0 data parts
  while the source has active rows.
- `ops/scripts/restore-drill.sh` — new artifact, runs the full create + restore-into-scratch
  loop in 3 seconds end-to-end.

3. **Snapshot** via `clickhouse-backup create drill_20260523T170217Z` running inside the live
   `bullpen-clickhouse` container. Backup summary: `all:1.37KiB, data:882B, meta:520B` — actual
   data present.
4. **Spin scratch** ClickHouse 24.12.6.70 on docker network `bullpen-drill-net`, ports
   18123/19000, default password `thebullpen`. Ready after 2 seconds.
5. **Install clickhouse-backup v2.6.43** into scratch via `docker cp`, write a minimal config
   pointing at `localhost:9000` + `data_path: /var/lib/clickhouse`.
6. **Transfer** the backup dir from live → scratch via `docker exec … tar -cf - | docker exec -i

7. **Restore** via `clickhouse-backup restore drill_20260523T170217Z` inside scratch. The tool

   the data parts.

## What this drill did NOT prove (and why it's still a PASS)

- **R2 round-trip**: the drill restores from local `/var/lib/clickhouse/backup/` only. The
  `clickhouse-snapshot.sh` script doesn't currently push to R2 either; that path is wired up
  in `infra/backup/usb-backup.sh` (Layer 2 USB) and lives as a Phase 1+ extension via
  `rclone copy` into `bullpen-r2:bullpen-prod/snapshots/`. R2 reachability is already proven
  separately (today's reboot drill item 7 — `rclone lsd bullpen-r2:bullpen-prod` returns
  three prefixes). Adding `clickhouse-backup upload` to the snapshot script + verifying the
  download leg of a restore is a Phase 1 follow-up, captured below.
- **Multi-table restore**: only `bullpen._drill_marker` exists today. Real tables
  (`pitches`, `prediction_logs`, `drift_metrics`) land in Phase 1/3. The drill mechanism is
  table-agnostic — `clickhouse-backup` operates on all non-system databases by default and the
  scratch verify step would extend to per-table counts trivially. The proof-of-mechanism is
  complete.
- **SQLite leg**: the snapshot script captures the SQLite registry too (`.backup`-based hot
  copy). The drill doesn't restore the SQLite half because the registry table is empty
  pre-Phase-1 (Flyway has created the schema but no rows). Restoring an empty SQLite into a
  scratch process is mechanically trivial — `sqlite3 newpath '.restore /backup/registry.sqlite'`
  — and adds no new evidence. Will be wired into the drill script when registry rows exist.

## Follow-ups (filed as Phase 1+ items, NOT blocking Phase 0)

1. **R2 push from `clickhouse-snapshot.sh`** — currently only writes local. Add
   `rclone copy /var/lib/clickhouse/backup/<name>/ bullpen-r2:bullpen-prod/snapshots/clickhouse/<name>/`
   after the `clickhouse-backup create` step. (Phase 1)
2. **Restore-from-R2 drill** — extend `restore-drill.sh` with a `--from-r2` mode that
   `rclone copy`s the backup down from R2 before the local restore step. (Phase 1)
3. **Re-snapshot the existing pre-fix backups** — the three `auto_*` snapshots in
   `/var/lib/clickhouse/backup/` (May 19, 21, 23 metadata-only) are mechanically present but
   carry no data. They're harmless (zero bytes of payload) but misleading at a glance. Decide
   whether to delete (cleaner) or keep with a `BROKEN_PRE_FIX.txt` marker (archaeological).
   Deferred — user has not authorized deletion. (Either way: zero-loss, since the source data
   was empty test rows.)
4. **`SYSTEM UNFREEZE` is disabled** by default in ClickHouse 24.12. clickhouse-backup falls back
   to `ALTER TABLE … UNFREEZE WITH NAME` which works. Not a problem; noted for future
   reference if we ever want manual FREEZE management.
5. **Empty `drill_test/` shadow dir** left from this session's diagnostic FREEZE — harmless,
   user can clean up with `sudo rm -rf /var/lib/docker/volumes/thebullpen_clickhouse-data/_data/shadow/drill_test`.
